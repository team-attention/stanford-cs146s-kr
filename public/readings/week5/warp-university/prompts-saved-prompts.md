# 저장된 프롬프트로 재사용 가능한 액션 실행하기

[영상 바로가기](https://youtu.be/pE15zjJmB4E)

## 핵심 요약

- **저장된 프롬프트**: Warp의 프롬프트 기능을 사용하면 반복 작업을 자동화하고 팀과 공유할 수 있음
- **Git 워크플로우 자동화**: 커밋 생성, 코드 리뷰, PR 생성을 프롬프트로 실행 가능
- **팀 협업**: Warp Drive를 통해 팀원 모두가 동일한 프롬프트를 사용할 수 있음
- **빠른 실행**: `Command + P`로 커맨드 팔레트를 열어 프롬프트를 빠르게 검색 및 실행
- **에이전트 모드 통합**: 저장된 프롬프트는 Warp의 에이전트 모드와 결합하여 컨텍스트 인식 작업 수행
- **코드 품질 개선**: 푸시 전 코드 리뷰 프롬프트를 실행하여 버그를 조기에 발견할 수 있음

## 전체 번역

안녕하세요, Zach입니다. 오늘은 Warp의 프롬프트(prompts)라는 기능을 사용하여 PR과 Git 커밋 작업을 할 때 워크플로우를 어떻게 간소화하는지 보여드리겠습니다. 이 기능의 정말 멋진 점은 많은 시간과 타이핑을 절약할 수 있다는 것입니다. 그리고 이 프롬프트는 저만 사용할 수 있는 게 아니라 우리 팀 전체가 사용할 수 있습니다. 그럼 바로 보여드리겠습니다.

여기 진행 중인 PR이 있다고 해봅시다. `git diff`를 실행해서 무슨 일이 일어나고 있는지 볼 수 있습니다. 미리 말씀드리자면, Warp에서도 곧 이 diff를 볼 수 있는 기능이 나옵니다. 자, 이제 괜찮은 지점에 도달했으니 커밋을 하고 싶고, 이 커밋에 합리적인 커밋 메시지를 넣고 싶습니다. `git commit`을 입력하기 시작할 수도 있지만, 긴 메시지를 직접 타이핑하는 건 별로 하고 싶지 않습니다. Warp의 에이전트 모드가 저를 위해 이 커밋을 만들어주길 원합니다. Warp에게 직접 요청할 수도 있지만, 더 빠른 방법이 있습니다. 바로 여기 Warp Drive 패널에 저장된 프롬프트를 만들어두었습니다. 한번 열어서 보여드리겠습니다.

이 프롬프트는 설명과 함께 커밋을 생성하고 origin에 푸시합니다. 커밋을 어떻게 만들지에 대한 지침도 제공합니다. 선택적으로 인자를 받을 수도 있지만, 저는 커밋 메시지를 자동으로 생성하도록 하겠습니다.

이걸 실행하려면 여기로 와서 클릭할 수도 있지만, 실제로는 `Command + P`를 눌러서 커맨드 팔레트를 열고 저장된 프롬프트를 검색해서 실행하는 게 더 빠릅니다. 이게 실행되는 동안 다른 멋진 기능들도 보여드리겠습니다.

이 프롬프트는 Warp Drive에 있기 때문에, 우리 팀의 Warp Drive 섹션에 속해 있습니다. 그래서 팀의 누구나 이 프롬프트를 사용할 수 있습니다. 누가 만들었는지, 마지막으로 언제 실행됐는지도 볼 수 있습니다. 오늘 실행됐네요. 그리고 이걸 실행하면 일반적인 에이전트 모드 쿼리로 실행됩니다. 즉, Warp 에이전트가 하는 모든 작업을 수행한다는 뜻입니다. 제 상태를 확인하고, diff를 보고, 제가 원하는 대로 커밋을 생성해줄 것입니다. 이 메시지는 여기서 일어나는 일을 정확하게 설명하고 있습니다. 그리고 origin에 푸시해주는데, 이게 바로 제가 일반적으로 하는 방식입니다.

이제 다른 저장된 프롬프트들도 보여드리겠습니다. Pull request를 만들기 전에 실행하는 프롬프트가 있습니다. 이 프롬프트는 에이전트에게 제 코드를 리뷰하고 문제가 없는지, 개선할 점이 있는지 확인하도록 요청합니다. 이 프롬프트를 실행해보겠습니다. "현재 브랜치의 코드 변경사항을 리뷰하고 로직이나 스타일 오류, 기타 개선사항이 있는지 확인해줘"라고 요청합니다. 이 방식이 GitHub에 푸시해서 리뷰하는 것보다 훨씬 빠릅니다. 빠른 피드백을 받는 멋진 방법이죠. 이 PR에 문제가 있는지 볼게요. 사실 아직 실행해보지 않았습니다. 어떤 결과가 나올지 모르겠네요. 아마 문제가 있을 수도 있습니다. 재미있는 건 에이전트가 실제로 이 코드를 작성했다는 점입니다. 그래서 문제가 있다면 스스로 고칠 수도 있겠네요.

로직 버그가 있다고 하네요. 흥미롭습니다. 실제로 버그처럼 보이네요. 놀랍습니다. 매우 똑똑하네요. 실제로 버그를 잡아냈습니다. 잠재적인 레이스 컨디션을 발견했습니다. 좋아요, 그럼 이건 아직 pull request를 만들 준비가 안 된 것 같네요. 코드 품질 문제가 있는 것 같습니다.

다시 말하지만, 에이전트가 실제로 이 코드를 작성했기 때문에 이런 결과를 보는 게 흥미롭습니다. 마지막 프롬프트 하나를 더 보여드리겠습니다. 제 전체 워크플로우가 어떤지 보여드릴게요. 이런 버그들이 없다고 가정해봅시다. 실제로는 지금 에이전트에게 이 버그들을 고쳐달라고 요청할 것입니다. 하지만 버그가 없다고 가정하면, 제가 프롬프트를 사용하는 또 다른 정말 멋진 기능은 실제로 PR을 여는 것입니다.

그래서 "내 변경사항으로 PR 만들기"를 찾아보겠습니다. 이것도 제가 항상 사용하는 프롬프트 중 하나입니다. 이걸 실행하고 싶다면 그냥 실행하면 됩니다. 짠! 이해하셨죠? 정말 좋습니다.

이건 당연히 Git 작업에만 국한되지 않습니다. 반복 가능한 모든 작업에 대해 저장된 프롬프트를 만들고 공유하고 사용할 수 있습니다. 팀 전체와 공유하고, 언제 실행됐는지 확인하고, 편집하고, 매개변수화할 수 있습니다. 정말 멋진 기능입니다. 오, 이건 유용하네요.
