# Warp에서 이미지를 컨텍스트로 사용하기

[영상 바로가기](https://www.youtube.com/watch?v=_Pc7bL0zAoM)

## 핵심 요약

**이미지 컨텍스트 기능의 가치**

- **텍스트의 한계 극복**: UI 컴포넌트나 시각적 요소는 텍스트로 설명하기 어렵지만, 이미지로는 즉시 전달 가능
- **"백문이 불여일견"의 실현**: AI 에이전트도 인간처럼 이미지를 통해 더 정확하게 이해

**실제 사용 사례: Figma → Warp 구현**

- **작업**: MCP 마켓플레이스 UI를 Figma 목업에서 실제 코드로 구현
- **프로세스**: Figma 스크린샷 → 에이전트 모드에 첨부 → 음성/텍스트로 지시 → 자동 코드 생성
- **결과**: 1-2일 걸릴 작업을 20분 만에 완료

**기술적 구현**

- **반복적 diff 적용**: AI는 큰 변경보다 작은 diff를 여러 번 적용하는 것이 더 효과적
- **자동 승인 모드**: 작업 초기에만 확인하고 나머지는 자동화 가능
- **토큰 최적화**: 클라이언트 측에서 이미지 크기를 지능적으로 조정하여 토큰 소비 최소화

**핵심 인사이트**

- 에이전트 상호작용 실패는 주로 "시작 단계"에서 발생 (잘못된 파일, 작업 오해 등)
- 이미지 컨텍스트는 UI 작업에 특히 효과적 (픽셀 단위 설명 불필요)
- 이 기능 자체도 Warp 에이전트로 개발됨 (dogfooding의 실례)

## 전체 번역

인간은 다섯 가지 감각을 가지고 있습니다.

보통 컴퓨터와 상호작용할 때 주로 텍스트를 사용하지만, 텍스트로는 잘 전달되지 않는 것들이 있습니다. "백문이 불여일견"이라는 말이 있죠.

저는 이 말이 매우 진실이라고 생각합니다. 특히 소비자 대상 제품이나 기업용 제품을 만들 때는 대부분 UI 컴포넌트가 포함되기 마련입니다.

>> 테이크 원 마크.

>> 저는 Adwight입니다. Warp의 소프트웨어 엔지니어로 에이전트 팀에서 여러분이 알고 사랑하시길 바라는 AI 기능들을 개발하고 있습니다. 오늘은 이미지 컨텍스트에 대해 이야기하려고 합니다. 이것은 Warp 내에서 특정 에이전트 쿼리에 이미지를 첨부하여 작업을 완료할 수 있게 해주는 기능입니다. 에이전트에게 "이건 몇 픽셀 정도 어긋나 있어" 같은 걸 설명하거나 특정 컴포넌트를 찾으라고 말하는 건 정말 어렵습니다.

그래서 이런 복잡한 설명을 하는 대신, 이제 그냥 이미지를 제공하면 됩니다. 그러면 훨씬 더 잘 이해할 수 있다고 생각합니다. 왜냐하면 결국 AI는 인간의 데이터로 학습되기 때문입니다. 인간이 이미지를 더 잘 이해하고 직관적으로 받아들이는 것처럼, 에이전트도 이미지를 통해 더 쉽게 이해하고 작업을 완수할 수 있습니다. 저는 Figma 목업에서 Warp로 무언가를 가져오는 것을 데모하고 싶었습니다. 이것은 엔지니어로서 제가 자주 하는 작업입니다. 저희의 훌륭한 디자이너들과 함께 작업하고 있죠. 특히 많은 분들이 MCP에 대해 들어보셨을 텐데, MCP 서버 설치를 위해 고려하고 있는 아이디어 중 하나를 보여드리고 싶습니다. 에이전트 모드에 대해 자세히 들어가기 전에 제품 컨텍스트를 조금 설명하겠습니다. 현재 Warp에는 MCP 서버를 위한 이런 인터페이스가 있습니다. 여기에서 특정 JSON과 세부 정보를 포함해서 MCP 서버를 추가할 수 있습니다. 하지만 저희가 도입하고 싶었던 것 중 하나는 MCP 서버를 위한 더 쉬운 설치 플로우였습니다. 그래서 저희 디자이너 Peter가 MCP 서버를 설치하는 더 시각적인 방법을 보여주는 멋진 목업을 만들었습니다. 특히 인기 있는 MCP 서버들을 위해 마켓플레이스라는 개념을 도입한 것이죠. 제가 이것을 구축하는 방법은 간단합니다. 이것의 스크린샷을 찍은 다음, 에이전트 모드에 넘겨서 무엇을 할 수 있는지 확인하면 됩니다. 이것을 컨텍스트로 첨부해봅시다. 이미지를 첨부하려면 여기로 가서 아이콘을 클릭하면 됩니다. 스크린샷으로 가보겠습니다.

좋습니다. 이제 첨부된 것을 볼 수 있습니다.

"이 MCP 마켓플레이스 컨셉을 구축하는 걸 도와줘." 음성을 사용할 수도 있는데, 조금 더 빠르니까요. 그래서 여기서 음성으로 전환해보겠습니다. "하드코딩된 데이터만 사용하고 싶어. UI 컴포넌트에 집중하자." 그리고 파일이 어디 있는지도 알려줄게요.

좋습니다. 그리고 쉽게 찾을 수 있도록 파일을 알려줍시다. 렌더링 로직이 이 collection rs 파일에 있다는 걸 알고 있습니다.

네, 이제 파일 구조를 살펴보고 있네요.

좋습니다. 이 이미지가 전달되었다는 걸 인식했고 이제 이 diff를 만들고 있습니다. 좋아요. 이게 적용하고 있는 첫 번째 diff입니다. 적용해봅시다. 참고로, diff를 편집하거나 다듬고 싶다면 이 코드 diff를 사용해서 할 수 있습니다. 이제 적절한 데이터 구조를 만들고 있습니다. 지금은 백엔드에 연결하거나 MCP 자체를 생성하지 않고, 그냥 UI, 즉 프론트 엔드 UI만 만들 겁니다. 좋습니다. 제가 전달한 이미지에서 컴포넌트들을 식별한 것을 볼 수 있습니다. 브라우저 자동화, 캘린더 관리, 클라우드 플랫폼 등이 보이네요. 그리고 저희 코드베이스에서 가져온 다양한 아이콘들도 있습니다. 말이 되네요. 변경사항을 적용하겠습니다. 플랜을 만들었고 정의한 할 일 목록을 진행하고 있습니다. 이 경우 특정 목표가 있죠. 우리가 도달하려는 목업과 일치시키는 것입니다. 다시 목업으로 돌아가면, Linear, GitHub, Stripe가 있고 여기에 몇 가지 세부사항이 있었던 걸 볼 수 있습니다. 이것은 이미지에서 기대한 것과 정확히 일치합니다. 그리고 진행하면서 원한다면 빠른 모드를 활성화할 수도 있습니다.

여기 아래에서 에이전트 모드 코드 diff를 자동 승인할 수 있습니다. 제가 준 작업에 대해 잘 이해하고 있고, 제가 제공한 컨텍스트를 잘 파악하고 있는지 확인하고 싶습니다. 제가 과거에 본 흔한 함정은 잘못된 파일을 편집하려고 하거나 작업을 실제로 잘못 해석하는 경우입니다.

보통 에이전트 상호작용이 실패하는 경우는 시작 단계에서이고, 나머지는 대부분 실행 과정입니다. 이제 저희 UI 프레임워크 로직을 사용해서 모든 렌더링 코드를 추가하고 있습니다. 코드 diff 뷰를 보면 GitHub UI와 유사하게 구조화되어 있어서 익숙할 겁니다. 추가하고 제거하는 코드 줄 수를 보여주고 실제 diff들도 볼 수 있습니다. 여러 diff가 있는 경우... 좋은 예시를 찾아보죠. 네, 여러 헝크(hunk)를 가질 수도 있습니다. 여기서 반복적으로 수행하고 있네요. 여기서 한 가지 참고할 점은, 일반적으로 AI 모델이 더 작은 diff를 수행하는 데 더 뛰어나다는 것을 발견했다는 겁니다. 그래서 반복적으로 diff를 적용하는 것이 AI 에이전트에게 모든 것을 한 번에 시키는 것보다 낫습니다. 한 번에 하면 일반적으로 최상의 결과가 나오지 않습니다. 하지만 원한다면, 예를 들어 여러 파일에 걸친 의미론적 작업이 있고 여러 줄의 코드가 있다면, 파일 간에 여러 헝크를 제안할 수도 있습니다. 거의 끝난 것 같습니다.

에이전트 모드가 방금 한 일에 대한 간단한 요약을 제공했습니다. 이것은 에이전트가 무엇을 했는지 확인하는 유용한 방법입니다. 여기서 UI 컴포넌트들을 볼 수 있습니다. 지금은 하드코딩된 데이터를 추가하고 있는데, 나중에 데이터베이스와 서버 엔드포인트 등에 연결할 수 있습니다.

이것을 실행해서 Warp가 무엇을 만들었는지 봅시다. 좋습니다. 이게 Warp입니다. MCP 서버 탭이 열려 있네요. 정말 멋집니다. 이게 Warp가 만든 것입니다. 여기서 우리가 기대했던 세 개의 MCP 서버, Linear, GitHub, Stripe를 볼 수 있습니다. 다음 단계는 실제로 이것을 백엔드에 연결하는 것입니다. 서버 엔드포인트에 연결하고, 기존 MCP 로직에 연결해서 이 워크플로우를 엔드 투 엔드로 활성화하는 거죠. 클릭 핸들러를 연결하고 모든 파이프라인을 구축할 겁니다. 하지만 UI에 어려움을 겪는 사람으로서, 예를 들어 과거에 세세한 CSS 디테일 작업을 별로 좋아하지 않았는데, 이것은 정말 놀랍습니다. 제가 하루나 이틀 걸렸을 지루한 작업을 에이전트가 약 20분 만에 완료했으니까요. 이것은 정말 재미있게 만든 기능이었습니다. 개인적으로 Warp에서 한동안 원했던 기능이었죠. 재미있는 사실 중 하나는 실제로 이 기능을 에이전트와 함께 Warp를 사용해서 만들었다는 겁니다. 하지만 가장 어려웠던 부분 중 하나는 이미지가 정말 빠르게 토큰을 소비할 수 있다는 점이었습니다. 그래서 제품 측면에서 생각해봤을 때, 사용자들이 의도치 않게 토큰 수를 폭발적으로 늘리는 위험을 주고 싶지 않았습니다. 우리가 추가한 것 중 하나는 실제로 클라이언트 측에서 이미지를 서버로 전달하기 전에, 그리고 우리가 지원하는 다양한 LLM 제공자에게 전달하기 전에 지능적으로 크기를 조정하는 것이었습니다. 여기서 우리가 하고 싶었던 중요한 것은 사용자들을 위해 이것을 더 효율적으로 만들고 이미지 컨텍스트를 위해 사용하는 토큰 양을 최소화하는 것이었습니다.
