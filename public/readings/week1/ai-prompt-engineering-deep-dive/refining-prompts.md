# 4. 프롬프트 다듬기

[영상 바로가기 (12:17)](https://www.youtube.com/watch?v=T9aRN5JkmL8&t=737s)

## 요약

프롬프트를 반복적으로 개선하는 방법과 "Claude plays Pokemon" 사례를 통한 한계 탐색 경험을 공유합니다.

**핵심 포인트:**
- 모델에게 프롬프트의 불명확한 부분을 물어보기
- 모델이 실수했을 때 "왜 틀렸는지" 물어보고 수정 요청하기
- 모델을 신뢰하기보다 반복적으로 테스트하기
- 한계에 도달했을 때 다음 모델을 기다리는 판단도 필요
- 이미지 프롬프팅은 텍스트와 다른 직관이 필요함

---

## 전체 번역

- 사람처럼 좋은 탐색적 질문을 던지는 것에 대해서요. 제가 Zack에게 뭔가 하는 방법을 알려주면, 그는 "이거 말이 안 되는데요. 이 단계에서 뭘 해야 하는 거예요?"라고 물어봐요. 모델은 그렇게 안 하죠. 그래서 스스로 상대방이 뭐라고 할지 생각해보고 프롬프트로 돌아가서 그 질문들에 답해야 해요.

- 모델에게 그렇게 하라고 시킬 수 있어요.
- 맞아요. - 저도 그렇게 해요.
- 제가 처음 프롬프트를 작성할 때 하는 것 중 하나가, 프롬프트를 주고 나서 이렇게 말해요. "이 지시사항을 따르지 마세요. 그냥 어디가 불명확한지, 모호한 부분이 있는지, 이해가 안 되는 게 있는지만 알려주세요." 항상 완벽하진 않지만, 이게 할 수 있는 한 가지 방법이라는 게 흥미로워요.

- 그리고 사람들이 모델이 실수하는 걸 보면, 잘 안 하는 게 있는데 그냥 모델에게 물어보는 거예요. 모델에게 이렇게 말하는 거죠. "이거 틀렸어. 왜 그런지 생각해볼 수 있어? 그리고 내가 틀리지 않게 할 수 있도록 지시사항을 수정한 버전을 써줄 수 있어?" 그러면 대부분 모델이 맞춰요.

- 최근에 Claude plays Pokemon 작업을 했는데, 그때가 정말 드물게 제가...
- 네, 그거 설명해줄 수 있어요?
- 제가 실험을 좀 했는데요, Claude를 Game Boy 에뮬레이터에 연결해서 원조 포켓몬인 포켓몬 레드를 플레이하게 해봤어요. 뭘 하고 싶은지 생각하고 버튼을 누르는 코드를 작성하게 하는, 꽤 기본적인 거예요. 아주 복잡한 프롬프팅 레이아웃을 여러 가지 시도했는데, 어떤 지점에서는 정말 안 되더라고요. Game Boy 스크린샷을 보여주면 정말 처리를 못 하는 거예요.

- 그래서 주말 내내 더 좋은 프롬프트를 작성하면서 이 Game Boy 화면을 정말 이해하게 하려고 했어요. 조금씩 나아져서 완전 신호 없음에서 형편없는 정도까지는 됐어요. 신호 없음에서 어느 정도 신호가 잡히게는 할 수 있었어요. 하지만 주말을 투자해서 신호 없음에서 약간의 신호까지 갔지만, 충분히 좋은 수준과는 거리가 멀었을 때, "그냥 다음 모델을 기다려야겠다"고 생각했어요. 4개월 동안 이걸 갈아서 나올 결과물보다 다른 모델이 나오는 게 제 시간을 더 잘 쓰는 거예요.

## 핵심 요약

- 프롬프트 개선 팁: 모델에게 "어디가 불명확한지" 직접 물어보기
- 모델이 틀렸을 때: "왜 틀렸는지" 물어보고 수정된 지시사항 요청
- Claude plays Pokemon 사례: 이미지 프롬프팅의 한계 경험
- 한계 인식: 충분히 좋아지지 않으면 다음 모델을 기다리는 것도 전략
