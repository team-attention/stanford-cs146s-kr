# 모델 컨텍스트 프로토콜(MCP): 개발자를 위한 종합 안내서

[원본 링크](https://stytch.com/blog/model-context-protocol-introduction/)

---

## 요약

"모델 컨텍스트 프로토콜(MCP)은 AI 모델과 외부 데이터 및 서비스를 연결하는 개방형 표준으로, 대규모 언어 모델(LLM)이 일관되고 안전한 방식으로 구조화된 API 호출을 수행할 수 있게 합니다."

MCP는 AI 애플리케이션과 외부 서비스 사이의 범용 어댑터 역할을 하며, 각 도구나 API마다 커스텀 통합 코드를 작성할 필요를 없앱니다.

---

## 모델 컨텍스트 프로토콜(MCP)이란?

MCP는 LLM과 다양한 데이터베이스, 애플리케이션, API 사이의 다리 역할을 하며, 복잡한 커스텀 코딩 없이도 원활한 통합을 가능하게 합니다. 클라이언트-서버 아키텍처를 사용하는데, AI 애플리케이션이 MCP 클라이언트를 실행하는 호스트 역할을 하고, 외부 통합 기능은 함수, 데이터 리소스, 프롬프트 템플릿 같은 기능을 노출하는 MCP 서버로 동작합니다.

---

## MCP의 가치

주요 이점은 다음과 같습니다:

- **빠른 도구 통합:** "MCP를 사용하면 각각을 처음부터 커스텀 코딩하지 않고도 새로운 기능을 연결할 수 있습니다." MCP 서버는 애플리케이션과 외부 도구 또는 데이터 소스 사이의 경량 중개자 역할을 합니다.

- **자율 에이전트:** MCP는 에이전트가 CRM에서 데이터를 수집하고, 이메일을 보내고, 데이터베이스 기록을 남기는 작업을 자연스러운 흐름으로 연결하는 다단계 워크플로우를 가능하게 합니다.

- **설정 부담 감소:** "애플리케이션이 MCP를 지원하면, 단일 메커니즘을 통해 얼마든지 많은 서비스에 연결할 수 있습니다."

- **일관성과 상호운용성:** "모델의 출력(함수 호출)과 도구 결과 모두 통일된 JSON 구조로 전달됩니다."

- **양방향 컨텍스트:** MCP는 도구와 함께 프롬프트(사전 정의 템플릿)와 리소스(데이터 컨텍스트)를 지원하여, 일회성 질의를 넘어서는 풍부한 상호작용을 가능하게 합니다.

---

## MCP 아키텍처

### 클라이언트-서버 구조

- **MCP 클라이언트:** AI 애플리케이션(챗봇, IDE 어시스턴트, 에이전트)에 내장됨
- **MCP 서버:** 표준화된 JSON-RPC 메시지를 통해 외부 기능을 노출함

### JSON-RPC 요청 예시

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/list",
  "params": {}
}
```

서버 응답에는 이름, 설명, 입력 스키마를 포함한 구조화된 도구 목록이 담깁니다.

---

## MCP 서버 구축 및 배포

개발자는 stdout으로 출력하거나 HTTP 엔드포인트를 제공할 수 있는 어떤 프로그래밍 언어로든 MCP 서버를 만들 수 있습니다. MCP는 확장성과 유연성을 제공하는 클라이언트-서버 아키텍처를 따릅니다. 로컬 개발 환경, 클라우드 플랫폼, 온프레미스 인프라 등 다양한 배포 옵션이 있으며, Cloudflare는 원격 MCP 서버 배포를 위한 강력한 기능을 제공합니다.

---

## MCP 클라이언트와 도구

MCP 클라이언트는 외부 데이터 소스나 도구에 접근하기 위해 서버에 연결하는 애플리케이션으로, Python, JavaScript, Java 등 다양한 언어로 구축할 수 있습니다. MCP 도구는 특정 기능을 제공하며, Claude Desktop(채팅 인터페이스)과 Cursor(플러그인 시스템)가 대표적인 예입니다. 커스텀 구현을 통해 특화된 데이터 분석이나 다중 서버 통합 시나리오도 지원할 수 있습니다.

---

## MCP와 다른 접근 방식 비교

| 특성 | MCP | 커스텀 통합 | ChatGPT 플러그인 | LangChain |
|------|-----|------------|-----------------|-----------|
| 통합 속도 | 빠름, 플러그 앤 플레이 | 느림, 커스텀 코드 필요 | 보통, 독점적 | 보통, 커스텀 필요 |
| 인증 | OAuth 표준 | 수동 API 키 관리 | 플러그인별 OAuth | 다양함 |
| 상호작용 유형 | 지속적, 컨텍스트 풍부 | 일회성 | 단발성 | 컨텍스트 제한적 |
| 개방형 표준 | 예 | 아니오 | 아니오 | 프레임워크 의존적 |

### 기존 접근 방식

**커스텀 통합:** 각 서비스마다 커스텀 코드를 작성하고 API 키를 수동으로 관리해야 하므로 노동 집약적입니다. MCP는 이러한 상호작용을 플러그 앤 플레이 시스템으로 중앙화합니다.

**ChatGPT 플러그인:** OpenAI가 2023년에 선보인 플러그인 시스템은 API 접근을 표준화했지만, 독점적이고 대부분 단발성 호출에 그쳤습니다. MCP는 지속적인 양방향 상호작용과 함께 "개방적이고 범용적인" 접근을 제공합니다.

**LLM 도구 프레임워크:** LangChain은 개발자 중심의 표준을 제공했지만, 동적 도구 발견 기능이 부족했습니다. MCP는 다양한 환경에 적용 가능한 프로토콜 수준의 표준화를 통해 이를 공식화합니다.

## MCP 실전: 기술 심층 분석

### 단계별 흐름

**1. MCP 서버에 연결**
호스트 애플리케이션이 클라이언트를 초기화하고 로컬 프로세스나 HTTP 스트림을 통해 연결을 수립하며, 프로토콜 핸드셰이크를 위한 초기화 메시지를 전송합니다.

**2. 사용 가능한 도구/리소스 탐색**
클라이언트가 서버 기능을 조회합니다:

```javascript
const tools = await mcpClient.request({ method: 'tools/list' }, ListToolsResultSchema);
```

**3. LLM의 도구 선택**
사용자가 "AAPL 주식의 최신 가격이 얼마야?"라고 질문하면, LLM은 `get_current_stock_price(company="AAPL", format="USD")`를 호출해야 한다고 판단합니다.

**4. MCP를 통한 도구 호출**
클라이언트가 tools/call 요청을 전송합니다:

```javascript
const result = await mcpClient.request({
    method: 'tools/call',
    params: { name: toolName, arguments: toolArgs }
}, CallToolResultSchema);
```

**5. LLM에 결과 반환**
애플리케이션이 도구 출력을 응답에 통합하고, 모델은 검색된 정보를 활용하여 사용자 질의에 계속 답변합니다.

### Claude 통합 예시

```javascript
// 1. 사용 가능한 도구와 함께 사용자 프롬프트를 LLM에 전송
const response = await anthropicClient.complete({
  prompt: "User: Can you list my projects?\nAssistant: ",
  model: "claude-3.5",
  tools: tools
});
for (const msg of response.messages) {
  if (msg.type === 'tool_use') {
    // 2. LLM이 도구 사용을 결정함
    const { name, args } = msg;
    // 3. MCP를 통해 도구 호출
    const toolRes = await mcpClient.request({
      method: 'tools/call',
      params: { name, arguments: args }
    });
    // 4. 도구 결과를 주입하고 계속 진행
    await anthropicClient.send({
      role: 'system',
      content: `Tool result: ${toolRes.result}`
    });
  } else {
    // 5. 일반 LLM 응답 처리
    console.log("Assistant:", msg.content);
  }
}
```

"MCP는 도구 실행을 위한 표준화된 호출/응답 계층을 제공하여", 다양한 LLM 벤더에서 유지보수 가능하고 강력한 AI 시스템을 구축할 수 있게 합니다.

## 초기 한계: 인증 공백

2024년 후반에 MCP가 등장했을 때, 원격 서버를 위한 표준화된 인증이 없었습니다. 초기 구현에서는 서버를 로컬이나 신뢰할 수 있는 환경에서 실행해야 했습니다. 많은 서버가 시작 설정을 통한 수동 자격 증명 제공을 가정했는데, 이는 다중 사용자 애플리케이션에서 확장성 문제를 야기했습니다.

"MCP는 개방적이고 인터넷 기반으로 설계되었지만, 인증 표준이 없어 안전한 원격 사용이 제약을 받았습니다."

## OAuth 2.0 인증 흐름

MCP는 안전하고 확장 가능한 원격 연결을 위해 OAuth 2.0을 채택했습니다:

1. **동적 클라이언트 등록(DCR):** 클라이언트가 OAuth 서버에 자동으로 등록되어, 수동 설정과 하드코딩된 자격 증명이 필요 없습니다.

2. **자동 엔드포인트 탐색:** 표준화된 메타데이터 URL을 통해 클라이언트가 OAuth 엔드포인트를 자동으로 발견하여 설정 부담을 줄입니다.

3. **안전한 인가와 토큰 관리:** 클라이언트가 사용자 권한에 맞게 범위가 지정된 OAuth 토큰을 획득하여, 명시적으로 허용된 리소스에만 접근하도록 보장합니다.

4. **확장 가능한 다중 사용자 지원:** OAuth 2.0은 본질적으로 다수의 동시 사용자와 서비스를 지원하며, 이는 클라우드 도입에 필수적입니다.

## 디버깅과 문제 해결

MCP는 개발자가 서버를 테스트하고 검사할 수 있는 대화형 디버깅 도구인 **MCP Inspector**를 제공합니다. 이 도구는 클라이언트-서버 상호작용에 대한 상세한 뷰를 제공하여 문제를 체계적으로 파악할 수 있게 합니다. 종합적인 디버깅 가이드에서 일반적인 문제와 해결책을 설명합니다.

## 실제 활용 사례

MCP는 다양한 구현을 가능하게 합니다:

- **AI 기반 챗봇:** 외부 데이터 소스와 도구에 접근하여 정확하고 최신 정보 제공
- **AI 기반 워크플로우:** AI 모델을 외부 시스템과 통합하여 복잡한 프로세스 자동화
- **향상된 AI 모델:** 외부 도구 및 데이터 소스와 상호작용하여 정확도 향상
- **AI 기반 자동화:** 금융, 의료, 제조 분야의 업무를 자동화하여 생산성 향상

"MCP는 AI 애플리케이션의 가능성을 열어, 개발자가 더 통합적이고 자율적이며 확장 가능한 솔루션을 만들 수 있게 합니다."

---

## 결론

MCP는 개발자가 지능형 언어 모델을 기존에 접근하기 어려웠던 소프트웨어와 데이터 생태계에 연결할 수 있게 하는 중요한 발전입니다. 공통 프로토콜을 도입함으로써, 팀은 커스텀 플러그인을 개발하지 않고도 "더 통합적이고 자율적이며 확장하기 쉬운 AI 시스템"을 구축할 수 있습니다.

프로토콜이 계속 발전하고 있지만, MCP나 유사한 접근 방식이 기본적인 AI 인프라가 될 것입니다. 개발자는 지금 MCP 개념에 익숙해지는 것이 좋습니다. "도구 통합의 '배관 작업'을 처리해주어 시간과 수고를 절약할 수 있기" 때문입니다.

"MCP는 AI 어시스턴트가 고립된 천재가 아니라, 잘 갖춰진 엔지니어이자 조수가 되는 세상을 가능하게 합니다. 다양한 시스템과 연동하고, 절차를 따르고, 필요에 따라 정보를 가져오거나 생성할 수 있으며, 이 모든 것이 통합되고 안전한 인터페이스를 통해 이루어집니다."
