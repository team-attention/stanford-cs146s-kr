# 효과적인 코드 리뷰 방법: GitHub 스태프 엔지니어의 철학

[원본 링크](https://github.blog/developer-skills/github/how-to-review-code-effectively-a-github-staff-engineers-philosophy/)

## 들어가며

GitHub 스태프 엔지니어 Sarah Vessels가 8년간 7,000건 이상의 풀 리퀘스트(Pull Request)를 리뷰한 경험을 바탕으로 코드 리뷰 접근법을 공유합니다. 그녀는 코드 리뷰가 소프트웨어 개발에서 가장 중요한 부분 중 하나라고 강조하며, 자신의 작업보다 팀원의 풀 리퀘스트 리뷰를 우선합니다.

## 코드 리뷰란?

풀 리퀘스트 리뷰를 통한 코드 리뷰는 협업자가 제안된 변경 사항에 코멘트를 달고, 수정을 승인하거나, 병합 전에 추가 개선을 요청할 수 있게 합니다. Vessels는 풀 리퀘스트를 "대화의 시작"으로 정의합니다. 작성자가 기존 코드 개선을 제안하는 것이죠. 리뷰어는 가정에 의문을 제기하고 다른 관점을 제공하여 작성자와 대화하면서 코드 품질을 높여야 합니다.

### 왜 코드 리뷰를 해야 하나요? 커리어에 도움이 됩니다

코드 리뷰는 지식 교환을 촉진하고 배포 속도를 높여 커리어에 큰 영향을 미칩니다. 리뷰는 동료와 관리자가 전문적 기여를 평가할 때 참조할 수 있는 "링크 가능한 산출물"을 만들어냅니다. 특히 복잡하거나 논쟁적인 변경을 다룰 때 효과적인 코드 리뷰 코멘트가 커뮤니케이션 능력을 보여줍니다. 훌륭한 리뷰는 제품 방향을 안내하고, 장애를 예방하며, 커리어 발전을 이끕니다.

## 코드 리뷰 프로세스 최적화하기

### 리뷰할 풀 리퀘스트 찾는 방법

Vessels는 GitHub 알림 인박스를 주요 작업 공간으로 삼아 CI 빌드를 기다리거나, 작업 사이, 또는 하루를 시작할 때 확인합니다. 팀들은 보통 리뷰 준비된 풀 리퀘스트를 공유하는 전용 Slack 채널을 운영하는데, 이것도 좋은 발견 채널입니다.

GitHub Slack 통합으로 `/github subscribe your/repo pulls +label:"your-team-label"` 같은 명령어로 관련 풀 리퀘스트를 채널에 구독할 수 있습니다.

그녀는 다음과 같은 고급 검색 쿼리 사용을 권장합니다: `is:open archived:false is:pr org:github -is:draft team-review-requested:github/relevant-codeowner-team`

이 쿼리는 조직 내에서 열려 있고, 아카이브되지 않았으며, 드래프트가 아니고, 특정 코드오너 팀이 리뷰어로 요청된 풀 리퀘스트를 찾습니다.

> "팀원이 이미 리뷰했더라도 풀 리퀘스트를 리뷰하고 싶어서 보통 review:required 검색 한정자를 생략합니다."

### 리뷰어 팀으로 알림 관리하기

대규모 리뷰어 팀은 책임 분산 문제를 일으킵니다. 모두가 다른 누군가가 리뷰할 거라고 생각해서 방치되거나 성급하게 병합되는 풀 리퀘스트가 생기죠. Vessels는 각 PR이 개인적으로 관련 있다고 느껴지도록 코드오너 팀 멤버십을 좁혀 알림을 관리 가능하게 유지하라고 권장합니다.

대규모 통합 팀이 대체 커버리지를 제공하지만, 기본 자동 리뷰어로 쓰면 안 됩니다. 코드 경계가 명확하게 정의된 잘 정리된 CODEOWNERS 파일이 알림을 제한하고 리뷰어 피로를 막는 데 도움이 됩니다.

한 가지 전략은 스케줄에 따라 멤버를 교체하는 자동화로 초기 대응 팀(first responder)을 만드는 것입니다. PagerDuty API로 현재 초기 대응자를 파악하고, Octokit 라이브러리로 자동으로 팀 멤버를 추가하고 제거할 수 있습니다.

### 자동화로 팀 전반의 코드 리뷰 표준화하기

CODEOWNERS 파일과 브랜치 보호 규칙 같은 저장소 수준 도구가 리뷰 표준을 강제합니다. 팀별 표준은 사람이 유지보수하고 명확히 문서화해야 합니다. 팀은 예상 처리 시간과 사용 중인 자동화를 포함해 코드 리뷰 프로세스를 문서화해야 합니다.

일부 팀은 들어오는 리뷰를 추적하는 프로젝트 보드를 씁니다. 외부 기여자가 수정하는 공유 API를 관리하는 팀에 효과적이죠. 다른 팀은 범위가 잘 지정된 코드 소유권과 규율 있는 리뷰 습관으로 GitHub 알림에 의존합니다.

외부 의존성이 많은 팀은 자동화된 코멘트로 기대치를 전달할 수 있습니다. 봇이 리뷰가 필요한 풀 리퀘스트에 피드백 예상 시점을 표시하는 메시지를 남길 수 있습니다.

## 좋은 코드 리뷰와 나쁜 코드 리뷰의 차이

좋은 코드 리뷰는 명확성을 더하고 코드를 개선된 상태로 이끕니다. 나쁜 리뷰는 구체성이 부족해 작성자가 철저함이나 다음 단계에 대해 확신을 갖지 못하게 합니다.

### 커뮤니케이션의 명확성

효과적인 리뷰어는 개인적 선호와 승인 차단 요소를 구분합니다. 같은 저장소에서 구체적인 예시를 제공하면 일관성을 보여주며 제안이 강화됩니다. 코멘트 없이 전면 승인하거나 거부하는 불명확한 리뷰는 작성자가 철저함에 의문을 갖게 합니다.

마찬가지로, 모호한 구현 일정은 작성자를 좌절시킵니다. 리뷰어는 제안이 승인을 막는 것인지 선택적 개선인지 명시해야 합니다.

### 효과적인 코드 리뷰 코멘트 예시

"새 메서드가 이 파일의 기존 스타일을 따라 [X]개의 매개변수를 받네요. 매개변수가 그렇게 많으면 가독성이 떨어지고 함수가 너무 많은 일을 한다는 신호입니다. 이 메서드와 기존 메서드들을 나중 풀 리퀘스트에서 리팩터링해서 매개변수 수를 줄이는 건 어떨까요?"

**이 코멘트의 강점:**
- 구체적인 세부 사항 제공
- 특정 코드나 이슈 참조
- 명확한 해결책 제안
- 근거나 설명 제시

### 리뷰 가능하지만 약한 코멘트 예시

**"이게 마음에 안 듭니다."**
- 개선: "이 줄이 너무 많은 일을 하네요. 단순화할 수 있을까요?"
- 개선: "n+1 쿼리 성능 문제가 있을 것 같습니다."
- 개선: "커스텀 구현 대신 [프레임워크의] 솔루션을 쓸 수 있을까요?"

**"이건 작동 안 할 거예요."**
- 개선: "[X] 때문에 작동 안 할 거예요. 이 이슈를 참고하세요: [링크]."
- 개선: "[PR 링크]에서 시도했는데 [X] 때문에 실패했어요."
- 개선: "[X] 문제가 생기면 [대안적 접근법]을 시도해 보세요."

**"이게 버그를 고친 것 같아요."**
- 개선: "[이슈 링크]를 고친 것 같아요."
- 개선: "[이슈 링크]의 버그를 고치는 건가요?"
- 개선: "[링크]의 버그 같네요. 수정 감사합니다!"

## 좋은 코드 리뷰를 하는 방법

### 질문하기

Vessels는 풀 리퀘스트 작성자가 자신의 변경에 대해 가장 많은 맥락을 가지고 있다고 봅니다. 다양한 기술 스택 경험에도 불구하고 작성자의 이해가 자신보다 낫다고 여기며 작성자의 답변을 신뢰합니다.

> "코드베이스나 팀이 처음이라면, 코드 리뷰에서 질문해서 기술 스택과 사용 가능한 내부 도구 및 라이브러리에 대해 배우세요."

코드 가정을 탐구하는 질문이 특히 가치 있습니다: 코드가 어떤 데이터 형태와 작동하나요? 불일치하는 형태를 잘 처리하나요? 리소스를 많이 쓰나요? 성능이 적절하게 유지될까요?

Vessels는 엣지 케이스 동작을 검증하는 자동화된 테스트를 포함한 답변을 선호합니다. 두 번째로 선호하는 것은 데이터 웨어하우스나 모니터링 시스템의 실증 데이터로 시나리오가 문제가 되지 않는 이유를 보여주는 것입니다.

### 긍정적 피드백 제공하기

질문 외에도 리뷰어는 풀 리퀘스트에서 좋았던 점에 코멘트해야 합니다. 이런 긍정적 피드백은 꼼꼼히 읽고 이해했음을 보여줍니다:

- "이 모듈의 다른 클래스에서 쓰는 패턴과 일치하네요."
- "테스트를 추가해 주셔서 감사합니다!"
- "이제 훨씬 더 명확하게 읽히네요."

이런 코멘트는 심리적 이점이 있습니다. 여러 제안을 받으면 코드 리뷰가 지칠 수 있습니다. 요구 없는 긍정적 코멘트는 동기 부여와 완료된 작업에 대한 인정을 제공합니다.

### 편견과 가정 인식하기

코드 영역이나 작성자 연차에 대한 리뷰어 편견은 문제를 간과할 위험이 있습니다. 경험 수준과 관계없이 누구나 실수합니다. 리뷰어의 관점이 배포 전에 문제를 잡아냅니다.

Vessels는 편견을 줄이기 위해 테스트 작성을 권장합니다. 테스트는 작성자 신뢰성에 의존하지 않고 객관적으로 동작을 검증합니다. 주니어 개발자도 명백한 주제에 대해 질문해야 합니다. 한 사람에게 불명확한 것은 다른 사람에게도 불명확합니다. 질문은 미래 독자를 위한 교육적 가치를 보존합니다.

### 승인 여부 결정

Vessels는 승인을 제품 개선을 막는 차단 게이트로 보기 때문에 신중하게 승인을 보류합니다. 풀 리퀘스트가 프로덕션을 깨뜨리거나 사용자에게 해를 끼치지 않는다면 승인을 보류하지 않고 선택적 변경을 제안할 수 있습니다. 작성자는 병합 전이나 후속 브랜치에서 피드백을 처리할 수 있습니다.

> "피처 플래그를 쓰고 풀 리퀘스트를 작게 만드세요. 변경이 덜 무서울수록 프로덕션에서 더 빨리 롤백할 수 있습니다."

리뷰할 때 피드백의 중요성을 고려하세요. 제안 처리를 위해 배포를 미루는 게 맞나요? 개선 사이클—작성자 수정, CI 대기, 재리뷰, 배포, 병합—이 그 변경에 가치가 있나요? 누락된 제안이 사용자 경험을 악화시키지 않는다면 구현 여부를 작성자에게 맡기세요.

'Request changes' 옵션은 리뷰어가 다시 승인할 때까지 병합을 막습니다. Vessels는 팀 판단을 신뢰하며 거의 사용하지 않습니다. 병합 전 발견이 걱정되는 즉각적인 보안 문제에만 'Request changes'를 선택합니다.

### AI 코딩 도구 사용 시 코드 리뷰가 중요한 이유

AI 코딩 도구는 부정확하거나 안전하지 않은 코드에 대한 안전장치를 포함하지만, 잘못된 안심을 만들면 안 됩니다. 인간 리뷰어가 최종 방어선이며 모든 코드를 동일한 주의로 리뷰해야 합니다.

## 코드 리뷰에서 최대한 얻는 방법

### 자신의 코드 리뷰하기

GitHub 시니어 소프트웨어 엔지니어 Paul Smith가 Vessels에게 다른 사람의 리뷰를 요청하기 전에 셀프 리뷰를 가르쳤습니다. 작성자는 다른 사람의 코드에서 의문을 가질 만한 명확하지 않은 변경에 인라인 코멘트를 남겨야 합니다. 셀프 리뷰는 분할이 필요한 너무 큰 풀 리퀘스트를 식별합니다.

팁: [lerebear/sizeup-action](https://github.com/lerebear/sizeup-action)으로 풀 리퀘스트 복잡성과 크기를 나타내는 라벨을 자동으로 붙이세요.

### 병합 후 리뷰 환영하기

리뷰 완료 전에 풀 리퀘스트가 병합되더라도 Vessels는 여전히 나중의 피드백을 환영합니다. 병합 후 코멘트는 미래 독자가 무슨 일이 있었는지 이해하는 데 도움이 되는 흔적을 남깁니다. 그녀는 병합된 풀 리퀘스트 피드백을 병합 전 피드백과 똑같이 처리합니다—설명 코멘트, 추가 풀 리퀘스트, 또는 진행 중인 작업을 캡처하는 새 이슈로.

### 드래프트 풀 리퀘스트 사용하기

풀 리퀘스트를 드래프트로 만들면 리뷰 준비 상태를 알립니다. Vessels는 불완전한 작업이나 실패하는 CI를 나타내기 위해 드래프트 상태를 씁니다. 다른 사람에게도 똑같이 기대합니다: 드래프트는 작성자가 리뷰 준비가 안 됐다는 뜻입니다. 리뷰 준비 상태는 충분한 승인만 있으면 병합할 수 있다는 뜻입니다.

드래프트 상태는 불완전함을 뜻합니다. Vessels는 충돌 해결이나 피드백 처리 시 풀 리퀘스트를 드래프트로 표시합니다. '준비됨'으로 바꾸면 이전 리뷰어에게 재평가를 알립니다.

### 품위 있게 대하기

"식초보다 꿀로 더 많은 파리를 잡는다"는 말이 여기 적용됩니다. Vessels는 풀 리퀘스트 코멘트에 답장합니다—특히 동의하지 않을 때. 글로 답장 안 해도 동의에는 엄지 척 이모지로, 감사에는 하트 이모지로 반응합니다.

작성자는 코멘트로 상황을 파악합니다. 제안에 동의하지만 현재 PR에서 구현 안 하기로 했다면 그렇게 말합니다. 나중 브랜치에서 피드백을 처리할 때 링크와 원래 리뷰어를 크레딧하는 태그와 함께 돌아옵니다.

"이것은 [링크]에서 @누구누구의 피드백을 처리합니다"처럼 관련 풀 리퀘스트에 리뷰어를 태그하면 맥락과 인정을 제공합니다. 피드백 약속을 이행하면 미래에 편하게 승인할 수 있는 신뢰를 쌓습니다.

## 마무리

특히 AI 코드 생성 시대에 제품 품질에 대한 코드 리뷰의 중요성은 아무리 강조해도 지나치지 않습니다. 두 번째 눈의 리뷰로 수많은 버그가 잡히고 장애가 예방되었습니다. 코드 리뷰는 매일 리뷰하고, 프로세스를 개선하고, 자동화를 구축하는 데 시간을 투자할 가치가 있습니다.

> "개발자가 지금 풀 리퀘스트를 철저히 리뷰하는 게 이미 프로덕션에 배포된 문제를 나중에 처리하는 것보다 더 빠르고 덜 고통스럽습니다."

독자들은 리뷰 대기열을 확인하고 여기서 논의한 원칙을 적용해 보세요. 더 많은 안내가 필요하면 Mickey Gousset와 Joshua Johanning이 GitHub Community 토론에서 제공하는 "풀 리퀘스트 리뷰를 위한 5가지 팁"을 참고하세요.

## 핵심 요약

- 코드 리뷰는 지식 교환, 커리어 발전, 제품 품질 향상에 필수입니다
- 풀 리퀘스트를 찾으려면 GitHub 알림, Slack 통합, 고급 검색 쿼리를 활용하세요
- CODEOWNERS 파일을 잘 정리하고 리뷰어 팀을 작게 유지해 알림 피로를 막으세요
- 좋은 리뷰는 구체적이고, 근거를 대며, 명확한 해결책을 제안합니다
- 질문하고, 긍정적 피드백을 주며, 편견을 인식하세요
- 'Request changes'는 즉각적인 보안 문제에만 신중히 사용하세요
- AI 코딩 도구 시대에도 인간 리뷰어가 최종 방어선입니다
- 다른 사람의 리뷰 전에 먼저 자신의 코드를 리뷰하세요
- 드래프트 상태로 리뷰 준비 상태를 명확히 알리세요
- 피드백에 품위 있게 응답하고 약속을 이행해 신뢰를 쌓으세요
