# 명세는 죽었다, 명세 만세!

[원본 링크](https://blog.ravi-mehta.com/p/specs-are-the-new-source-code)

**저자:** Ravi Mehta, Danny Martinez
**게시일:** 2025년 7월 31일

## 서론

저자는 커리어 전반에 걸쳐 명세(specification)가 점점 짧아지는 것을 목격했습니다. Microsoft에서는 PM의 가치를 명세 두께로 측정했고, Tripadvisor에서는 전설적인 명세가 냅킨 위에 휘갈겨 쓴 것이었습니다. 하지만 팀들은 명세를 가치 있는 작업이 아닌 서류 절차 정도로 취급합니다. 엔지니어의 코드와 디자이너의 인터페이스는 칭찬하면서 명세 품질은 간과하죠.

이것은 근본적인 실수입니다. Ravi Mehta의 Product Competency Toolkit에서 Feature Specification은 12가지 역량 중 1순위로, 실행, 전달, 품질의 근간입니다. 훌륭한 명세야말로 효과적인 프로덕트 매니지먼트의 기반입니다.

## 변화하는 지형

개발 환경이 변하고 있습니다. 엔지니어들은 그 어느 때보다 빠르게 움직이고, AI가 대략적인 개념을 몇 분 안에 작동하는 코드로 바꿉니다. 병목이 '만드는 것'에서 '이해하는 것'으로 이동했습니다: **무엇을 만들어야 하는지 파악하고, 요구사항을 중심으로 팀을 정렬하는 것.**

한때 일시적인 서류에 불과했던 명세가 이제 프로덕트 매니지먼트의 기반입니다. 사실상 소스코드 그 자체가 된 것입니다.

## PM이 갑자기 병목이 된 이유

Andrew Ng(앤드류 응)은 최근 전례 없는 추세를 언급했습니다: "제 인생에서 처음으로 엔지니어보다 PM을 두 배로 늘려달라는 제안을 받았습니다."

이전 예측이 맞았습니다. AI가 엔지니어를 가속화하면서, 기업은 PM을 더 적게가 아니라 더 많이 필요로 합니다. 제품 개발이 빨라지면서 모든 PM 역량, 즉 고객 이해, 기능 설계, 임팩트 검증에 압박이 가해지고, 모든 수요가 하나의 산출물에 집중됩니다: 바로 명세입니다.

## 명세가 새로운 소스코드다

전통적인 프로그래밍에서 사람이 읽는 소스코드는 최적화된 기계어로 컴파일됩니다. 소스코드가 진정한 원천(source of truth)이며, 주석, 구조, 문서를 담고 있습니다.

OpenAI의 Sean Grove는 도발적인 논제를 제시합니다: 잘 작성된 프롬프트(명세)가 새로운 소스코드라는 것입니다. 현재 AI 개발은 이 논리를 거꾸로 적용합니다. 팀이 신중하게 프롬프트를 작성하고, AI가 코드를 생성하면, 팀은 프롬프트를 버리고 코드만 보존합니다.

"이건 소스를 파쇄한 다음 바이너리를 아주 신중하게 버전 관리하는 것과 같습니다"라고 Grove는 말합니다.

코드는 명세의 "손실 압축된 투영(lossy projection)"입니다. 코드를 읽어도 완전한 의도를 알 수 없습니다. 바이너리를 디컴파일해도 원래 주석이나 변수명은 복구되지 않습니다. 반면 명세는 모든 것을 담고 있습니다. 충분히 상세한 명세는 "훌륭한 TypeScript, 훌륭한 Rust, 서버, 클라이언트, 문서, 튜토리얼, 블로그 포스트, 심지어 팟캐스트"까지 생성할 수 있습니다.

중요한 점은, 명세가 사람과 기계 모두를 공유된 목표로 정렬한다는 것입니다. Grove는 말합니다: "문서화된 명세는 사람들을 효과적으로 정렬하며, 소통하고 토론하고 참조하고 동기화하는 데 사용하는 산출물입니다."

그의 예측: "가까운 미래에 가장 효과적으로 소통하는 사람이 가장 가치 있는 프로그래머가 될 것입니다. 말 그대로, 효과적으로 소통할 수 있다면 프로그래밍할 수 있습니다."

희소한 기술은 코딩이 아닙니다. 완전한 의도와 가치를 담은 명세를 작성하는 능력입니다. 프로덕트 매니저에게 이것은 원래부터 해온 일이며, 이제 기계도 그것을 경청합니다.

## 프로토타입이 명세를 대체하지 못한 이유

역사적으로 제품 생명주기는 명세에서 시작해 와이어프레이밍, 디자인, 프로토타이핑, MVP 개발 순으로 진행되었습니다. 필요하지만 부담스러운 과정이었죠.

이 패러다임이 근본적으로 바뀌었습니다: **명세는 이제 입력이 아니라 출력인 경우가 많습니다.**

v0, Lovable, Replit 같은 현대 도구로 엔지니어 없이도 몇 시간 안에 작동하는 프로토타입을 만들 수 있습니다. 단순히 더 빠른 게 아닙니다. 근본적으로 다릅니다. "바이브 코딩(vibe coding)"으로 만든 프로토타입으로 기능 명세를 작성하기 전에 실제 고객 피드백을 수집할 수 있습니다. 팀은 가정을 검증하고, 플로우를 반복하고, 인터랙션을 다듬습니다.

**기존 워크플로우:** 모호한 아이디어 → 와이어프레임 → 디자인 → 엔지니어가 만든 MVP → 고객 피드백 → 고통스러운 명세 수정 → 재설계 → 재구현 → 불확실성.

**새로운 워크플로우:** 모호한 아이디어 → 빠른 프로토타입 → 고객 피드백 → 명확한 명세 → AI 지원 구현.

프로토타입은 명세를 없앤 게 아닙니다. 명세를 더 좋게 만들었습니다.

## 실제 명세 중심 개발

decimals의 창업자 Danny Martinez가 실용적인 명세 중심 개발을 보여줍니다. 그의 플랫폼은 크리에이터 이코노미 전문가들이 네트워크 인재를 일자리에 연결하도록 돕습니다.

랜딩 페이지 출시 후, 팀은 전문가들이 자신의 페이지 링크에 빠르게 접근할 수 있도록 해야 했습니다. Martinez의 공동창업자가 메시지를 보냈습니다: "헤더에 company-apply 페이지로 이동하는 버튼이 필요해요. 저는 지금 이메일 작업에 집중하고 있어요. 이건 바이브 코딩 가능한 수준이니 시간 되면 해주세요."

**사용된 도구:**
- 프로젝트 관리: Linear
- IDE: VS Code
- 익스텐션: GitHub Copilot Pro
- 모델: Claude Sonnet 4
- MCP 서버: Linear MCP server

**프로세스 단계:**
1. Slack 메시지로 Linear 티켓 생성
2. 티켓에 원하는 문구 명확히 하기
3. Copilot 열고 Linear 티켓에 접근하도록 Claude에 프롬프트
4. 티켓을 코드베이스와 대조 검토하도록 Claude에 프롬프트
5. 브랜치 생성하고 변경사항 구현하도록 Claude에 프롬프트
6. 변경사항 테스트
7. GitHub PR 생성
8. 엔지니어 리뷰/승인 대기

핵심 인사이트: 비개발자가 전략적인 Claude 프롬프트만으로 Linear 티켓, 코드베이스, 엔지니어링 협업을 성공적으로 수행했습니다.

**필수 요건:**
1. **구체적으로 작성하기:** 모호한 명세는 지저분한 코드베이스를 만듭니다. Claude가 초기 티켓 초안을 다듬고 명세 가이드라인을 수립하도록 도움받을 수 있습니다.
2. **선별적으로 적용하기:** 이 접근법은 단순한 작업에 적합합니다. 복잡한 티켓은 경험 있는 인력이 필요합니다.
3. **관문 유지하기:** 엔지니어 리뷰로 단순함과 기능성 사이의 균형을 보장합니다.

규칙이 바뀌었습니다. 명세는 LLM을 포함해 제품을 만드는 모든 사람의 진정한 원천(source of truth)입니다.

적절한 설정만 있으면 비개발자도 코드베이스에 기여할 수 있습니다. 시스템을 점진적으로 이해하고, Claude에 계속 의존하지 않고 독립적으로 변경사항을 구현할 수 있게 됩니다.

더 나아가, AI 에이전트가 티켓을 처리하는 동안 크리에이터들은 커피 브레이크를 즐길 수도 있을 것입니다.

## 결론

William Gibson(윌리엄 깁슨)은 이렇게 말했습니다: "미래는 이미 여기에 있다. 다만 고르게 분배되지 않았을 뿐이다."

AI의 혜택은 여전히 심하게 불균등합니다. 코드, 텍스트, 이미지 생성은 "AI 속도"로 작동하며 양자 도약을 이뤘습니다. 반면 고객 대화, 니즈 발굴, 구매 설득은 여전히 "인간 속도"에 머물러 있습니다.

이러한 불균등한 분배가 제품 팀을 재편합니다. 초점이 구현에서 이해로 이동합니다. 전통적인 PM 기술, 즉 고객 니즈 파악, 명확한 문제 정의, 우아한 솔루션 설계가 기하급수적으로 가치가 높아졌습니다.

뛰어난 PM은 인사이트를 명세로 전환합니다: 팀을 정렬하고, 구현을 안내하며, 점점 자동화되는 개발 환경에서 지속적인 참고 자료가 되는 산출물로.

## 핵심 요약

- 명세의 위상이 "필요한 서류 절차"에서 "새로운 소스코드"로 격상되었습니다
- AI가 엔지니어를 가속화하면서 PM이 새로운 병목이 되었고, 명세 작성 능력이 핵심 역량이 되었습니다
- 잘 작성된 프롬프트(명세)가 코드의 진정한 원천이며, 코드는 명세의 "손실 압축된 투영"에 불과합니다
- 프로토타입 도구(v0, Lovable, Replit)가 워크플로우를 바꿨지만, 명세를 대체하지 않고 오히려 더 좋게 만들었습니다
- 비개발자도 명확한 명세와 AI 도구로 코드베이스에 기여할 수 있으며, 엔지니어 리뷰가 품질을 보장합니다
- 미래의 가장 가치 있는 프로그래머는 가장 효과적으로 소통하는 사람이 될 것입니다
