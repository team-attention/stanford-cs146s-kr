# 코딩 에이전트 101: 실제로 일을 완수하는 기술

[원본 링크](https://devin.ai/agents101)

## 서론

이 가이드는 "제품에 구애받지 않는", "전술적", "기술적" 특성을 지향하며, 2025년 코딩 에이전트(coding agent)와 효과적으로 협업하려는 엔지니어를 대상으로 합니다. 고객 경험과 Cognition 팀이 Devin을 개발하면서 얻은 교훈을 바탕으로 작성되었습니다.

## 시작하기: 프롬프트 기초

### 무엇을 원하는지뿐 아니라 _어떻게_ 해야 하는지 말하라

일반적인 요청 대신 구체적인 방법론을 제시하세요. 가이드에서 설명하듯이 에이전트를 "의사결정이 불안정할 수 있는 주니어 코딩 파트너"로 대하세요.

**예시:** 단순히 유닛 테스트를 요청하는 대신, 테스트할 기능을 명시하고 엣지 케이스(edge case)를 식별하며 모킹(mocking) 요구사항을 명확히 하세요.

### 에이전트에게 시작점을 알려주라

리포지토리, 관련 문서, 핵심 컴포넌트를 명시하세요. 이렇게 하면 불필요한 시행착오를 줄일 수 있습니다.

**예시:** "[여기] 최신 문서를 확인하고 model groups 디렉토리에 새 구현 파일을 만들어줘."

### 방어적 프롬프팅을 실천하라

새로 들어온 인턴에게 하듯이 혼란을 줄 수 있는 부분을 미리 예측하고 사전에 명확히 설명하세요.

**예시:** "주의해, 코드를 변경할 때마다 테스트 전에 바인딩을 다시 컴파일해야 할 거야."

### CI, 테스트, 타입, 린터에 접근 권한을 부여하라

테스트 도구와 타입 체커를 통한 강력한 피드백 루프가 에이전트 성능을 향상시킵니다. 가이드에 따르면 팀은 "대부분 타입이 없던 Python SDK에서 전부 타입이 있는 SDK로" 전환했다고 합니다.

### 당신의 전문성을 활용하라

인간의 감독은 여전히 필수입니다. "궁극적으로 코드의 최종 정확성에 대한 책임은 당신에게 있습니다."

## 워크플로우에 에이전트 활용하기

### 새 작업을 즉시 처리하라

주요 업무에 집중하면서 빠른 조사나 변경 사항은 에이전트에게 위임하세요. 많은 팀이 "버그 수정이나 사소한 기능 업데이트를 논의할 때 Slack에서 간단히 @Devin을 태그합니다."

### 이동 중에도 코딩하라

모바일 접근으로 출장이나 출퇴근 중에도 긴급한 이슈를 처리할 수 있습니다.

### 잡일을 넘기라

커밋 이분법 탐색(bisecting)이나 문서 업데이트 같은 반복 작업을 위임하세요.

### 분석 마비를 건너뛰라

에이전트에게 여러 아키텍처 접근법을 구현하게 하여 실제로 비교해볼 수 있게 하세요.

### 프리뷰 배포를 설정하라

각 PR마다 자동으로 프리뷰 배포를 생성하도록 CI/CD 파이프라인을 구성하세요.

## 중급: 더 큰 티켓 위임하기

### 초안 작성을 자동화하라

상당한 규모의 작업에서 에이전트가 초기 PR 초안을 만들 수 있습니다. 가이드는 이렇게 비유합니다: "자율 에이전트는 초기 계획을 바탕으로 작업을 시작하고 초안 솔루션을 만듭니다."

현실적인 기대치는 "완전한 자동화가 아닌 약 80%의 시간 절감"입니다.

### PRD를 함께 개발하라

복잡하거나 모호하게 정의된 작업의 경우, 에이전트와 함께 상세한 계획을 세우세요. 에이전트가 "우리 인증 시스템은 어떻게 동작하나요?" 같은 탐색적 질문을 조사할 수 있습니다.

### 체크포인트를 설정하라

여러 부분으로 구성된 작업에서는 명확한 중단점을 설정하세요: "계획 → 청크 구현 → 테스트 → 수정 → 체크포인트 리뷰 → 다음 청크"

### 에이전트가 스스로 작업을 검증하도록 가르쳐라

테스트 프로세스를 명확히 설명하여 에이전트가 향후 작업을 독립적으로 검증할 수 있게 하세요.

### AI 핫스팟의 테스트 커버리지를 높이라

중요한 번역이나 리팩터링을 위임하기 전에 자주 수정되는 영역의 테스트를 강화하세요.

## 고급: 워크플로우 자동화

### 반복 작업을 위한 단축키 만들기

일반적인 자동화 후보로는 피처 플래그(feature flag) 제거, 의존성 업그레이드, 기능 PR에 대한 테스트 생성 등이 있습니다.

### 지능적인 코드 리뷰 및 규칙 적용

에이전트가 일부 경우 기존 린트 규칙보다 일반적인 실수를 더 정확하게 잡아낼 수 있습니다.

### 인시던트와 알림에 연결하라

에이전트는 API나 CLI 연동을 통해 자동으로 트리거될 수 있습니다. 하지만 가이드는 프로덕션 이슈에 대해 "AI의 디버깅 능력은 그다지 뛰어나지 않다"고 경고합니다. 종단 간 수정을 시도하기보다는 의심스러운 오류를 표시하는 정도로 활용하세요.

## 커스터마이징 및 성능 개선

### 환경 설정

언어 버전, 의존성, 사전 설정된 인증 등 팀 구성과 정확히 일치하도록 에이전트 환경을 맞추세요.

### 커스텀 CLI 도구와 MCP를 만들기

Linear 티켓 정보 가져오기나 개발 환경 재시작 같은 반복 워크플로우를 위한 스크립트를 만드세요.

### 에이전트의 지식 베이스에 추가하기

프레임워크별 패턴, 프로젝트 아키텍처, 테스트 컨벤션, 권장 도구에 대한 피드백을 체계화하세요.

## 실무 고려사항

### 자율 에이전트의 한계

**제한된 디버깅 능력:** 완전한 수정을 기대하기보다 가능한 근본 원인을 요청하세요.

**미세한 시각적 추론 부족:** 픽셀 단위로 Figma 목업을 정확히 구현하기를 기대하기보다 재사용 가능한 컴포넌트가 있는 디자인 시스템을 활용하세요.

**지식 컷오프:** 에이전트는 사전 학습된 지식에 의존하므로 새 라이브러리의 경우 "명시적으로 최신 문서를 가리켜 주세요."

### 시간 관리 및 손실 최소화

**일찍 손절하라:** 근본적인 오해 징후가 보이면 상호작용을 중단하세요.

**실험을 다각화하라:** 다양한 프롬프트를 시도하고 성공적인 작업 유형에 더 집중하세요.

**새로 시작하라:** 막혔을 때 "에이전트의 경우 사람보다 새로 시작하는 것이 정답인 경우가 훨씬 많습니다."

### 보안 및 권한 관리

- 일회용 이메일로 전용 에이전트 계정 생성
- 개발/스테이징 환경만 제공
- 가능하면 읽기 전용 API 접근 권한 부여
- 외부 서비스와 상호작용하는 스크립트는 사람이 직접 실행

## 결론

가이드는 발전하는 기능에도 불구하고 "소프트웨어 엔지니어는 사라지지 않을 것"이라고 강조합니다. 자동화가 개인의 영향력을 증폭시키고 병렬 작업 관리를 가능하게 하더라도 "프로젝트, 시스템, 코드에 대한 진정한 오너십은 그 어느 때보다 중요합니다."

## 핵심 요약

- **프롬프트 기초**: 무엇을 할지뿐 아니라 어떻게 할지 구체적으로 지시하고, 에이전트를 주니어 파트너처럼 대하며 방어적으로 프롬프팅하라
- **워크플로우 통합**: 빠른 작업 위임, 이동 중 코딩, 반복 작업 자동화에 에이전트를 활용하라
- **중급 활용**: 초안 자동화로 80% 시간 절감을 목표로 하고, PRD 공동 개발과 체크포인트 설정으로 대형 작업을 관리하라
- **고급 자동화**: 반복 작업 단축키, 지능적 코드 리뷰, 인시던트 알림 연동을 구축하라
- **한계 인식**: 디버깅 능력과 시각적 추론의 한계를 인지하고, 일찍 손절하며 막히면 새로 시작하라
- **보안**: 전용 계정, 스테이징 환경, 최소 권한 원칙을 적용하라
- **핵심 메시지**: 자동화가 발전해도 엔지니어의 오너십과 최종 책임은 여전히 중요하다
