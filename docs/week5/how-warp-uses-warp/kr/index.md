# Warp가 Warp를 만들기 위해 Warp를 사용하는 방법

## 코딩 의무 규정

*Warp의 코딩 의무 규정: 모든 코딩 작업은 Warp로 시작하라*

1. 모든 코딩 작업은 Warp에서 프롬프트로 시작한다.
2. 코딩 작업이 성공하면 좋다! #warped-it 채널에 성공 사례를 공유하라.
3. 10분이 지났는데도 프롬프트 작성에 시간을 허비하고 있다면...
    1. 무엇이 실패했는지 관련 #feedback- 채널에 피드백을 공유하라
    2. 프롬프트, 대화 ID 등을 함께 공유하여 문제 발생 지점을 파악하고 이를 평가(eval) 케이스로 만들 수 있게 하라
    3. 이는 팀원들이 Warp 활용 팁을 서로 공유할 수 있는 기회이기도 하다
4. 그 다음에는 다른 AI 코딩 도구(예: Cursor, Claude)를 사용해 보고 더 나은 결과가 나오는지 확인하라 – 만약 그렇다면, 무엇이 효과적이었고 무엇이 그렇지 않았는지 보고하라.
5. 마지막으로, 그냥 손으로 코딩하라...우리는 실제로 소프트웨어를 출시해야 하고, 그것이 여전히 가장 빠르다면 그렇게 해야 한다.

*Warp의 코딩 의무 규정: "왜"*

1. Warp에서 프롬프트로 코딩하면 실제로 많은 작업이 더 빠르며 전체 엔지니어링 생산성이 높아진다.
    1. 코드를 직접 작성하는 시간을 절약한다
    2. 여러 작업을 동시에 병렬로 진행할 수 있다
    3. 익숙하지 않은 코드 영역에서 작업할 때 특히 유용하며, 코드베이스가 커질수록 그런 영역이 많아진다 – Warp는 코드베이스의 새로운 부분을 빠르게 학습하는 데 큰 도움이 된다
2. 우리 제품을 직접 사용하는 것(dogfooding)이 제품 개선의 최선책이다.
    1. 이 의무 규정의 주요 이유 중 하나는 우리가 만드는 것을 직접 사용하고 피드백을 제공하는 것이다.
    2. 우리는 사용자들에게 이런 방식으로 일하라고 요청하고 우리 비즈니스를 여기에 걸고 있으므로, 우리 자신도 기꺼이 사용할 만큼 이 워크플로우를 신뢰해야 한다
3. 경쟁 제품을 사용하면 우리 제품의 개선점과 가능성에 대한 직관이 생긴다.

## Warp로 코딩하기 위한 가이드라인

### 리뷰를 위해 제출하는 코드에 대한 책임은 당신에게 있다

- 리뷰를 위해 제출하는 모든 코드를 직접 작성한 것과 동일한 수준으로 **반드시** 이해해야 한다
- 에이전트 모드로 생성한 모든 코드는 **반드시** 직접 작성한 코드와 동일한 품질이어야 한다...
    - 잘 구조화되어야 한다
    - 잘 테스트되어야 한다
    - 주석, API, 임포트, 테스트 파일 위치 등 우리의 코딩 컨벤션을 따라야 한다
    - 기타 등등...
- 버그나 낮은 코드 품질을 "AI가 작성했다"로 변명할 수 없다 – AI는 도구이지, 우리 소프트웨어 팀의 책임 주체가 아니다
- 프롬프트로 코딩할 때도 일반적인 How We Code의 모든 가이드라인이 여전히 적용된다 (작은 PR, 피처 플래그 등)

### 에이전트에게 원하는 결과만 말하지 말고, 변경 사항의 엔지니어링 방법을 말하라

- 다시 말해, 에이전트에게 "X처럼 동작하는 기능을 원한다"라고만 말하면 에이전트는 우리 기준에 맞지 않는 코드를 생성할 가능성이 높다.
- 대신, 에이전트에게 기능을 *어떻게* 만들어야 하는지 말하라
    - 데이터 모델은 어떤 형태여야 하는가
    - 원하는 API는 무엇인가
    - 코드는 어디에 위치해야 하는가
    - 어떤 테스트를 작성해야 하는가
- 에이전트가 결과만 목표로 탐색하게 내버려 두면 "동작"은 하지만 유지보수, 테스트, 조합이 불가능한 코드를 생성하여 많은 시간을 낭비하게 된다

### 코딩 작업을 더 작은 구성 요소로 분해하라 / 큰 변경을 원샷으로 처리하지 마라

- 에이전트에게 작고 독립적인 변경과 커밋을 요청하고 자주 올바른 방향으로 가고 있는지 확인하면 Warp로 더 효과적으로 코딩할 수 있다
- 에이전트가 많은 코드를 생성해야 하는 복잡한 변경을 원샷으로 처리하려는 것은 안티패턴이다. 원샷은 간단한 변경에는 괜찮지만, 복잡한 기능에는 적합하지 않다.
- 에이전트가 진행하면서 자주 테스트를 작성하게 하면 점진적 작업을 강제하는 좋은 방법이 된다.
- 진행 중인 변경 사항을 자주 확인하여 직접 작성했을 때처럼 진행되고 있는지 확인하라.
    1. 지금은 GitHub UI나 다른 PR 리뷰 도구를 사용할 수 있다
    2. 결국 Warp가 이 리뷰 플로우를 더 네이티브하게 지원할 것이다
- 필요할 때 되돌릴 수 있도록 Warp가 진행하면서 좋은 메시지와 함께 커밋하게 하라

### Warp가 무엇을 하는지 이해하고 있는지 확인하라

- Warp가 변경을 시작하기 전에 설명이나 계획을 요청하면 좋다
- 계획을 반복적으로 수정하고 변경 옵션과 다른 접근 방식을 요청하라. 차이점을 만들기 전에 큰 그림을 파악하라
- 반대로, Warp가 큰 변경을 처음부터 끝까지 수행하게 하여 무엇이 가능한지 볼 수 있지만, 우리의 품질 기준을 충족하기 위해 사후에 이해 가능한 조각들로 분해해야 할 것으로 예상하라.

### Warp 코드를 최대한 활용하기 위해 학습에 시간을 투자하라

- Warp를 더 많이 사용하여 코딩할수록 어떤 컨텍스트를 포함해야 하는지, 어떤 유형의 프롬프팅이 가장 효과적인지 등에 대한 직관이 쌓인다
- 이 지식을 팀과 공유하라 – 프롬프트로 코딩하는 것은 완전히 새로운 분야이며 우리 모두 잘 하는 방법에 대해 다른 직관을 가지고 있다. 함께하면 더 빨리 배운다.
- Warp의 고급 기능을 사용하여 워크플로우를 개선하라...
    - 규칙, 프롬프트, Warp Drive 객체를 영구적인 컨텍스트로 활용하라
    - 파일 태깅에 "@"를 사용하라
    - UI 변경 및 디버깅을 위해 이미지를 첨부하라
    - 다른 모델로 실험하라
- 효과적으로 여러 작업을 병렬로 진행할 수 있도록 warp-internal과 warp-server의 여러 복사본을 로컬에 설정하라

### 이슈를 보고하라

- 이것은 아무리 강조해도 지나치지 않다 – 무언가가 잘 작동하지 않으면 적절한 #feedback- 채널에 보고하라
- 이것은 다음을 의미할 수 있다
    - 서버 오류 또는 클라이언트 버그
    - 에이전트가 빙빙 돌면서 요청을 소모하고 시간을 낭비함
    - 에이전트가 컨텍스트를 잊어버림
    - 작업 달성에 도움이 될 누락된 기능
    - 높은 지연 시간, AI가 멍청하게 행동함, 잘못 적용된 diff 등등...
- 피드백 처리 측면에서, 이러한 이슈에 대한 우리의 첫 번째 본능은 **평가(eval) 케이스를 만드는 것**이어야 한다

### 컨텍스트를 쉽게 검색하기 위해 MCP 서버를 설정하라

- 우리 모두가 갖춰야 할 서버들...
    - Sentry
    - Linear
    - Notion
    - Slack
    - <그 외 무엇이 있을까>
- 에이전트 모드가 패닉을 수정하고 Linear에서 상황을 파악하는 데 얼마나 유능한지 놀랄 것이다.

### 반복되는 컨벤션을 위한 규칙을 설정하라

- 에이전트에게 반복적으로 지시하는 것들(예: 어떤 린터를 실행할지, unwrap / expect, 임포트, 테스트 위치 등 우리의 코드 컨벤션)에 대해 Warp Drive 규칙을 설정하는 데 시간을 투자하라

### 반복되는 워크플로우를 위한 프롬프트를 설정하라

- 작업을 반복하고 있다면, Warp Drive 프롬프트를 설정할 가치가 있다.
- 예를 들어, 나는 커밋 생성, PR 설명 열기 및 편집, 머지 충돌 해결 등에 프롬프트를 사용한다.

## 핵심 요약

**코딩 의무 규정의 핵심**
- 모든 코딩 작업은 Warp 프롬프트로 시작하며, 10분 후에도 진전이 없으면 피드백을 공유하고 다른 도구를 시도하거나 수동 코딩으로 전환한다
- 자사 제품을 직접 사용(dogfooding)하여 제품을 개선하고, 경쟁 제품 사용으로 직관을 키운다

**코드 품질 및 책임**
- AI가 생성한 코드도 직접 작성한 코드와 동일한 품질 기준을 적용해야 한다
- "AI가 작성했다"는 버그나 낮은 품질의 변명이 될 수 없다

**효과적인 에이전트 사용법**
- 원하는 결과뿐 아니라 *어떻게* 엔지니어링할지 구체적으로 지시하라
- 큰 변경을 원샷으로 처리하지 말고 작은 단위로 분해하여 점진적으로 진행하라
- 에이전트가 시작하기 전에 계획을 요청하고 큰 그림을 파악하라

**생산성 향상 팁**
- MCP 서버(Sentry, Linear, Notion, Slack)를 설정하여 컨텍스트 검색을 용이하게 하라
- 반복되는 컨벤션과 워크플로우를 위한 Warp Drive 규칙과 프롬프트를 설정하라
- 이슈 발생 시 피드백 채널에 보고하고, 평가(eval) 케이스로 구축하라
