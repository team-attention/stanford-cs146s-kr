# Warp로 에이전트 규칙 만들기

[영상 바로가기](https://www.youtube.com/watch?v=OyrpkeL6WNY)

## 핵심 요약

- **규칙(Rules) 기능**: Warp Drive의 기능으로 에이전트에게 프로젝트별 컨텍스트와 코딩 규칙을 제공
- **실제 사용 사례**: Docker 파일에서 패키지 설치 시 일관된 패턴(apt-get) 유지
- **반복 작업 개선**: 한 번 규칙을 설정하면 향후 세션에서도 자동으로 적용
- **자연어 컨텍스트**: 자연어로 요청하면 에이전트가 팀의 기존 문서와 노트북에서 관련 정보를 찾아줌
- **워크플로우**: 태스크 시작 → 에이전트에 자연어로 질문 → 첫 시도에서 패턴 불일치 발견 → 규칙 생성 → 재시도로 올바른 결과 획득

## 전체 번역

W by warp mark.

안녕하세요, 저는 Maggie입니다. Warp의 엔지니어이고 Warp에서 규칙(rules) 기능을 만든 엔지니어 중 한 명입니다. 이 기능은 기본적으로 Warp Drive의 여러 기능 중 하나로, 에이전트를 위한 더 많은 컨텍스트를 구축할 수 있게 해줍니다. 그래서 제가 현재 작업하고 있는 태스크 중 하나는 실제로 eval을 추가하는 것인데, 이는 제가 작업해온 다른 기능에 대한 여러 테스트 케이스 같은 것입니다. 그럼 에이전트 모드 eval을 어떻게 시작할 수 있을까요?

이제 에이전트가 말하고 있습니다. "Bessie, 우리가 몇 가지 노트북을 찾았어요. 설정 방법과 실행 방법에 대한 문서에 뭔가 있을 거예요." 제가 태스크를 시작할 때 이게 정말 도움이 됩니다. 아마도 이미 노트북을 작성한 팀원 중 한 명이 만든 것에서 나온 것 같습니다.

이번에는 Rust 구문 오류를 테스트하는 eval을 추가하고 싶습니다.

그럼 저는 "Rust를 포함하도록 Docker 파일을 업데이트하자"라고 말할 수 있습니다.

업데이트된 Docker 파일이 제가 실제로 원했던 방식과는 다르게 Rust를 설치하고 있습니다. GCC와 Python이 둘 다 이 apt-get을 통해 설치되고 있는 것을 볼 수 있습니다. 그래서 저는 실제로 이 변경사항을 stash하고 나서 그 관례를 더 따르도록 규칙을 만들 것입니다. 미래에 제가 더 많은 eval을 추가하고 더 많은 언어나 패키지를 추가하는 것을 볼 수 있습니다. 일회성으로 하기보다는 이것을 향후 세션에서 사용할 규칙으로 유지하는 것이 낫겠습니다.

제가 여기에 추가할 규칙은 "항상 apt-get을 사용하여 패키지를 설치하고, Python과 GCC를 설치할 때 사용된 패턴을 따른다"입니다.

이제 솔직히 저는 그냥 에이전트 모드에게 한 번 더 시도해보라고 요청할 것입니다. 이번에는 그 규칙을 따라서 제가 실제로 원하는 코드를 제공해주길 바랍니다. 그리고 Rust가 추가되고 있습니다. 제가 원하는 올바른 코드인 것 같습니다.

네, 그게 다입니다.

독특한 점은 제가 에이전트 모드에 자연어로 물어보기만 하면 컨텍스트를 가져올 수 있다는 것입니다. 그리고 그건 정말 놀라운 일입니다.
