# Warp로 Warp의 입력 만들기

[영상 바로가기](https://www.youtube.com/watch?v=ySzUj7kMZ64)

## 전체 번역

좋아요. 제가 바로 시작할까요?

[음악] >> 좋습니다. Dave, 테이크 원. 시작.

>> 안녕하세요, 저는 Dave입니다. Warp에서 제품 디자인을 이끌고 있습니다. 오늘은 유니버설 입력(universal input)의 디자인과 이를 Warp를 사용해 어떻게 만들었는지에 대해 이야기하려고 합니다. 이런 재설계에 접근할 때 가장 큰 과제는 디자인이 어떠해야 하는지에 대한 사람들의 기대를 다루는 것입니다. 입력은 터미널 작업의 주요 인터페이스이고 개발자들이 매일 하루 종일 사용하는 것이기 때문에, 처음부터 논쟁의 여지가 있을 것이고 무엇이어야 하는지에 대한 많은 다른 의견들이 있을 것이라는 걸 알고 있었습니다. 저희 팀의 제품 디자이너인 Peter와 저는 입력 디자인을 여러 번 반복 작업했습니다. 그래서 여기 Figma를 띄워놓고 우리가 작업하고 반복했던 디자인 파일들이 있습니다. 제 다른 화면에 유니버설 입력이 있는데, 기본적으로 이 디자인이 하는 일은 이전에 에이전트 모드에 숨겨져 있던 모든 에이전트 컨트롤을 전면에 가져오는 것입니다. Peter와 제가 이 입력의 디자인을 완성한 후, 내부적으로 판매했고 사람들은 대부분 그것에 흥분했습니다. 그러나 이를 구축할 시간이 되었을 때, 우리 엔지니어링 팀은 에이전트 모드의 응답 품질을 개선하고 에이전트 개발 환경을 준비하느라 꽤 바빴습니다. 그래서 이 디자인을 구축할 리소스가 실제로 충분하지 않았습니다. 그래서 저는 생각했습니다. "내가 그냥 Warp로 하면 어떨까?" 그래서 저는 Warp를 사용해 유니버설 입력을 만들기 시작했습니다. 이건 정말 간단한 예제인데, Warp 내부에 git diff 칩이 있고 이것은 어떤 git 브랜치에 있는지 표시하고 브랜치의 열린 변경사항도 표시합니다. 지금 보시면 딱 1픽셀 너무 높은데, 디자이너인 저에게는 이것이 많이 신경 쓰입니다. 너무 높은 이유는 실제 변경 표시의 폰트가 1픽셀 너무 크기 때문입니다. 그래서 이를 줄이기 위해 빠른 수정을 하려고 합니다.

Warp는 디자이너에게 매우 강력한데, Warp 자체와 같은 정말 복잡한 코드베이스를 이해할 수 있게 해주기 때문입니다. Warp는 Rust로 작성되었고 백만 줄 이상의 코드를 가지고 있습니다. 일부 파일은 2만 줄이나 됩니다.

완전히 커스텀한 UI 프레임워크를 처음부터 만들어서 사용합니다. 그래서 인터넷에 이를 어떻게 사용하거나 읽거나 디자이너로서 이해하는지에 대한 문서가 실제로 없습니다. 그래서 제가 가장 먼저 할 일은 Warp에게 이 코드가 어디 있는지 물어보는 것입니다. 저는 실제로 이 git 칩을 만들지 않았습니다. 우리는 이 프로젝트를 마무리하는 데 도움을 줄 엔지니어들을 데려왔고 그들이 만들었습니다. 그래서 저는 코드에 대해 그렇게 익숙하지 않습니다. 유니버설 입력의 git 칩이 코드 어디에 있는지 물어보면, 이제 Warp가 제 코드베이스를 검색하고 실제로 관련 코드를 찾을 것입니다. 유니버설 입력의 git 칩을 참조하는 여러 다른 파일들을 이미 가져온 것을 볼 수 있습니다. 그리고 이제 실제로 그 파일들을 검색하고 git 칩 표시를 제어하는 특정 함수들을 찾을 것입니다. 코드베이스 컨텍스트 시맨틱 검색과 전통적인 find 및 grep 도구의 조합을 사용합니다. 그리고 display_chip.rs 내부에서 주요 구현을 찾은 것을 볼 수 있습니다. 유니버설 입력 git 브랜치 칩과 관련 렌더링 메서드를 찾은 것을 볼 수 있습니다.

그리고 어디에 표시되는지 구성하는 파일과 다른 관련 파일들을 볼 수 있습니다. 그래서 이제 폰트 크기를 변경하도록 요청하겠습니다. git diff 변경 텍스트의 폰트 크기가 1픽셀 너무 큽니다.

1픽셀 줄여주세요.

그러면 이제 Warp가 display_chip.rs 파일을 살펴보고 폰트 크기의 인스턴스를 찾을 것입니다. 시스템 폰트 크기에서 1을 뺀 것으로 설정되어 있지만, 실제로는 -2로 설정하고 싶습니다. 그리고 이미 그것을 알아냈습니다. 제가 -2로 설정하도록 요청하고 있다는 것을요. 그리고 여기서 폰트 크기의 모든 인스턴스를 찾아 저를 위해 수정한 것을 볼 수 있습니다. 정말 멋지죠. 이제 애플리케이션을 빌드하고 작업을 확인하겠습니다. 다른 패널에서 터미널 모드로 전환하고 cargo run을 실행하는데, 이것이 로컬에서 애플리케이션을 빌드하는 방법입니다.

이것은 1분 정도 걸릴 것입니다.

좋습니다. 이제 애플리케이션이 빌드되었고 git 칩이 올바른 크기입니다. 여기서 나란히 비교해보면, 텍스트가 약간 더 작아졌고 이것이 칩의 높이를 올바른 크기로 만든 것을 볼 수 있습니다. 디자이너로서 저를 기쁘게 만드네요.

## 핵심 요약

### 주요 내용
- **유니버설 입력 재설계**: Warp의 제품 디자이너 Dave가 Warp AI를 사용해 Warp의 유니버설 입력 인터페이스를 재설계하고 구현한 사례
- **디자이너의 코드 작업**: 엔지니어링 리소스 부족으로 디자이너가 직접 Warp AI를 활용해 복잡한 코드베이스(Rust, 100만+ 줄)에서 작업
- **실제 사례**: Git diff 칩의 폰트 크기를 1픽셀 줄이는 작업을 Warp AI로 수행

### 핵심 기능
1. **코드베이스 탐색**: Warp AI가 자동으로 관련 코드 파일과 함수를 검색
2. **시맨틱 검색**: 코드베이스 컨텍스트와 시맨틱 검색을 결합하여 정확한 위치 파악
3. **자동 수정**: 자연어 요청("1픽셀 줄여주세요")을 코드 변경으로 자동 변환
4. **빠른 반복**: 디자이너가 엔지니어 도움 없이 UI 세부 조정 가능

### 시사점
- **디자이너-개발자 간극 해소**: AI 도구가 비개발자도 복잡한 코드베이스에서 작업 가능하게 함
- **생산성 향상**: 리소스 제약 상황에서도 제품 개선 지속 가능
- **내부 도구 활용(Dogfooding)**: Warp가 자사 제품을 활용해 자사 제품을 개선하는 사례
