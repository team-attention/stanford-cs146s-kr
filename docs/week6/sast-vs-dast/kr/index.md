# SAST vs. DAST vs. RASP: 애플리케이션 보안 테스팅 방법 비교

[원본 링크](https://www.splunk.com/en_us/blog/learn/sast-vs-dast.html)

## 소개

전 세계 정보 보안 지출은 **2025년까지 2,120억 달러**에 이를 전망이며, 2024년 대비 15% 증가한 수치입니다. Gartner는 생성형 AI와 클라우드 도입이 이러한 성장을 이끈다고 분석하며, 2027년까지 GenAI가 전체 사이버 공격의 17%에 관여할 것으로 예측합니다.

기업은 기존 접근 방식을 넘어서는 정교한 보안 전략이 필요합니다. SAST, DAST, RASP 같은 보안 솔루션은 다층적인 애플리케이션 보호를 제공합니다.

## SAST란?

정적 애플리케이션 보안 테스팅(Static Application Security Testing, SAST)은 화이트박스 보안 테스팅 방법론입니다. SAST는 애플리케이션이 실행되지 않는 상태에서 소스 코드나 바이너리를 분석해 보안 취약점을 찾아냅니다.

개발자는 SAST로 다음과 같은 보안 위험을 탐지합니다:
- 크로스 사이트 스크립팅(XSS)
- 안전하지 않은 역직렬화
- 버퍼 오버플로우
- OWASP 취약점

SAST는 취약점을 조기에 발견해 소프트웨어 개발 수명 주기의 핵심 구성 요소로 기능합니다. 조기 발견으로 수정 비용을 크게 절감할 수 있어, 개발자는 제품 출시 전에 코드를 반복적으로 테스트할 수 있습니다.

SAST는 자동화를 통해 CI/CD 파이프라인에 통합되어 "Secure DevOps" 또는 "DevSecOps" 실천을 가능하게 합니다. 자동화된 테스팅으로 확장할 수 있어 코드 수준 위험을 효율적으로 처리합니다.

## DAST란?

동적 애플리케이션 보안 테스팅(Dynamic Application Security Testing)은 블랙박스 테스팅 방식을 사용합니다. "동적(Dynamic)"이라는 용어는 **애플리케이션이 실행되는 동안** 보안을 평가한다는 의미입니다. SAST와 달리 DAST는 소스 코드 접근이 필요 없습니다.

DAST는 다음을 수행합니다:

1. 외부 공격자 행동 시뮬레이션
2. 런타임 취약점을 찾기 위한 외부 테스트

DAST가 식별하는 보안 결함:
- 서비스 거부(DoS) 취약점
- 안전하지 않은 서버 구성

실제 공격 시나리오를 모방해 DAST 도구는 정적 테스팅이 놓칠 수 있는 약점을 드러냅니다.

개발자는 보통 SDLC 후반, 배포 전에 DAST를 수행합니다. 라이브 환경에서 테스트하면 포괄적인 런타임 보안 평가가 가능합니다.

## SAST는 어떻게 작동하나요?

개발자는 SAST 도구로 사전 정의된 규칙과 탐지 방법(패턴 매칭, 데이터 흐름 분석 등)을 적용해 코딩 오류와 취약점을 찾습니다. 이 도구는 IDE나 CI/CD 파이프라인에 통합되어 코딩과 테스팅 단계에서 자동 스캔을 수행합니다.

### 일반적인 SAST 방법

**패턴 매칭**은 취약한 암호화 알고리즘이나 안전하지 않은 API 호출 같은 알려진 취약 코딩 패턴을 코드베이스에서 찾습니다.

**데이터 흐름 분석**은 애플리케이션 전체의 데이터 이동을 추적해 신뢰할 수 없는 입력이 민감한 작업으로 이어지는 SQL 인젝션이나 버퍼 오버플로우 같은 취약점을 찾습니다.

**제어 흐름 분석**은 애플리케이션의 제어 구조(루프, 조건문)를 검사해 경쟁 조건 같은 논리적 결함이나 취약점을 발견합니다.

**사용자 정의 규칙 생성**으로 개발자는 코딩 모범 사례를 규칙화합니다:
- SQL 쿼리에서 정제되지 않은 입력 표시
- 적절한 이스케이핑 없이 브라우저에 출력되는 사용자 입력 식별
- 취약하거나 폐기된 함수 강조 (예: JavaScript의 `eval()`, C의 `strcpy()`)
- 하드코딩된 API 키와 패스워드 탐지
- 입력 검증 누락 인스턴스 표시

**의존성 스캐닝**은 서드파티 라이브러리와 프레임워크를 분석해 의존성 취약점을 찾습니다.

**의미 분석**은 구조뿐 아니라 코드의 의미를 해석해 안전하지 않은 구성이나 API 오용을 탐지합니다.

**머신 러닝 모델**은 데이터셋 전반의 패턴을 분석해 이전에 알려지지 않은 취약점을 인식하는 ML 알고리즘을 활용합니다.

### SAST 도구의 성능 고려사항

SAST 도구는 특히 대규모 코드베이스 분석 시 상당한 CPU와 메모리를 요구합니다. 백만 줄 코드의 전체 스캔은 수 기가바이트 RAM을 소비할 수 있습니다.

스캔 시간은 코드베이스 크기에 따라 크게 달라서, 소규모 프로젝트는 몇 분, 엔터프라이즈 애플리케이션은 몇 시간이 걸립니다. 이는 다음에 영향을 줍니다:
- 빌드 파이프라인 일정
- 개발자 생산성

변경된 코드만 분석하는 증분 스캐닝을 지원하는 최신 SAST 도구를 고려하세요. 규칙 선택과 범위 정의를 통한 성능 튜닝도 중요합니다. 잘못 구성된 도구는 핵심이 아닌 코드 경로까지 불필요하게 분석하기 때문입니다.

## DAST는 어떻게 작동하나요?

DAST 도구는 입력 필드에 악성 데이터를 전송하는 것 같은 실제 공격을 시뮬레이션해 애플리케이션 동작과 응답의 취약점을 찾습니다.

### DAST 단계

**1단계: 스캐닝** - 웹 애플리케이션을 스캔해 진입점(URL, 폼, API)을 발견하고, 애플리케이션 구조를 매핑하며 잠재적 공격 표면을 식별합니다.

**2단계: 공격 시뮬레이션** - 크로스 사이트 스크립팅과 크로스 사이트 요청 위조 같은 취약점을 테스트하기 위해 조작된 요청을 전송합니다.

**3단계: 취약점 탐지** - 애플리케이션 응답을 분석해 보안 약점을 찾고, 공격 상황에서 애플리케이션이 예상대로 동작하는지 평가합니다(예: 악성 데이터 주입을 통한 SQL 인젝션 탐지).

**4단계: 보고** - 탐지된 취약점과 수정 권장 사항이 담긴 보고서를 생성합니다.

최신 DAST 솔루션은 AI 기반 분석과 실시간 데이터 통합 같은 고급 기능을 갖추고 있습니다. 자동으로 테스트 세트를 생성하고, 애플리케이션 구조에 동적으로 적응하며, 머신 러닝 알고리즘으로 오탐을 최소화합니다.

### DAST 도구의 성능 고려사항

성능 병목현상:

- 실행 중인 애플리케이션과의 **활성 상호작용**으로 애플리케이션 및 웹 서버의 CPU와 메모리 사용량이 일시적으로 증가
- **대량의 동시 요청**으로 상당한 네트워크 대역폭 소비
- DAST 스캐너가 고유한 요청으로 애플리케이션 캐시를 우회하는 **캐시 무효화**
- 애플리케이션 서버 메모리 사용량을 늘리는 **테스트 세션 관리**

이 문제를 최소화하려면:

- 피크 시간대를 피해 스캔 예약
- 스테이징 또는 프리프로덕션 환경에서 스캔 수행
- 스로틀링 기능으로 동시 요청 제어
- 스캐닝 중 애플리케이션 서버 스레드 풀 증가
- DAST 스캐너 IP에 별도 캐싱 정책 구성
- 성능 임계값 초과 시 자동 스캔 일시 중지 구현

## SAST vs. DAST: 주요 차이점

| 기준 | SAST | DAST |
|------|------|------|
| 테스팅 유형 | 화이트박스 테스팅; 소스 코드에 접근해 내부에서 테스트 | 블랙박스 테스팅; 소스 코드 접근 없이 외부에서 테스트 |
| 소프트웨어 유형 | 웹 애플리케이션, 웹 서비스, 씩 클라이언트 지원 | 주로 웹 애플리케이션과 웹 서비스 지원 |
| SDLC 단계 | SDLC 초기에 수행 | SDLC 후반, 배포 중 또는 배포 후 수행 |
| 탐지 취약점 | 코딩 결함: SQL 인젝션, XSS, 버퍼 오버플로우 등 | 런타임 문제: 서버 구성 오류, DoS 취약점, 애플리케이션 수준 결함 |
| 수정 비용 | 낮음; SDLC 초기에 발견 | 높음; 후반에 발견되어 긴급 수정이 필요한 경우 많음 |
| 런타임/환경 문제 | 런타임이나 환경 관련 취약점 탐지 불가 | 런타임이나 환경 요인으로 나타나는 취약점 식별 |
| 깊이 vs 너비 | 심층적인 코드 수준 취약점 인사이트 | 외부 보안 위험에 대한 광범위한 시야 |
| 도구 통합 | IDE 및 CI/CD 파이프라인에 자동 분석용으로 통합 | CI/CD 파이프라인에 지속적 런타임 테스팅용으로 통합 |
| 오탐/미탐 | 심층 분석으로 오탐이 높음 | 오탐은 낮지만 내부 가시성 제한으로 미탐 위험 높음 |
| 기술 의존성 | 프로그래밍 언어와 프레임워크에 의존; 도구 호환성 필요 | 애플리케이션 프레임워크와 독립적; 외부 상호작용 방식 |

## SAST와 DAST의 장단점

### SAST의 장점

- 소프트웨어 개발 수명 주기 초기에 취약점 발견
- 기본 함수와 복잡한 분기를 포함해 코드베이스 종합 분석
- 애플리케이션 실행 없이 소스 코드에서 작동
- 정확한 취약점 위치를 포함한 종합적인 인사이트와 실시간 피드백
- 대시보드로 추적 가능한 보고서 생성
- 수동 코드 리뷰보다 빠르고 효율적인 자동화 분석

### SAST의 단점

- 수동 검토가 필요한 오탐을 자주 생성
- 런타임이나 환경 특정 취약점(구성 오류) 식별 불가
- 프로그래밍 언어별 도구가 필요해 유지보수 복잡
- 외부 라이브러리, API, REST 엔드포인트 이해에 어려움

### DAST의 장점

- 애플리케이션 실행 중에만 나타나는 취약점 식별
- 외부에서 작동해 소스 코드 접근 불필요; 서드파티 애플리케이션과 컴파일된 코드에 효과적
- 실제 공격 시나리오 시뮬레이션으로 SAST가 놓치는 위험 식별
- 프로그래밍 언어와 독립적으로 작동
- 메모리 소비, 리소스 사용, 서드파티 인터페이스를 포함한 전체 시스템 평가

### DAST의 단점

- 외부 레이어에 집중해 깊은 취약점을 놓칠 가능성
- SDLC 후반에 적용되어 수정 비용과 시간 증가
- 소스 코드 분석으로만 식별 가능한 취약점 미탐지 (예: 세션 토큰용 `Math.random()` 같은 안전하지 않은 난수 생성기)
- 대규모 프로젝트에 맞춤 인프라 필요; 리소스 집약적

## SAST vs. DAST: 언제 사용해야 하나요?

SAST와 DAST는 SDLC 보안에서 상호 보완적 역할을 합니다. 적절한 타이밍을 이해하면 포괄적인 커버리지를 확보할 수 있습니다.

### SAST 사용 시점

**개발 초기 단계** - 개발자는 배포 전에 SQL 인젝션과 하드코딩된 자격 증명 같은 취약점을 찾아 수정 비용과 복잡성을 줄입니다.

*예시:* SAST 도구가 Python 코드를 스캔해 소스 코드에 포함된 하드코딩된 API 키를 표시합니다. 개발자는 환경 변수를 사용하도록 리팩터링합니다.

**코드 리뷰** - SAST는 버전 관리 시스템에 통합되어 커밋 전 코드 스캔을 가능하게 해, 안전한 코드만 저장소에 들어가도록 합니다.

*예시:* SAST 도구가 검증되지 않은 사용자 입력으로 파일 경로를 구성하는 경로 탐색 취약점을 탐지합니다. 개발자는 입력을 정제하고 안전한 라이브러리 함수를 사용합니다.

**CI/CD 파이프라인** - SAST는 CI/CD 프로세스에서 자동 스캔을 실행해 개발자에게 실시간 피드백을 제공합니다.

### DAST 사용 시점

**프리프로덕션 테스팅** - 개발자는 스테이징이나 QA 환경에서 안전하지 않은 구성, 인증 결함, 불충분한 접근 제어 같은 런타임 취약점을 찾습니다.

*예시:* DAST 도구가 민감한 데이터를 노출하는 데이터베이스 구성 오류를 탐지해, 개발자가 배포 전에 구성을 수정할 수 있습니다.

**배포 후 모니터링** - DAST 도구는 배포된 애플리케이션을 계속 스캔해 환경 변경이나 새로운 위협으로 인한 취약점을 찾아 지속적인 보안을 유지합니다.

**서드파티 통합 테스팅** - DAST는 외부 공격자 관점을 모방해 서드파티 API와 연결된 시스템의 취약점을 찾습니다.

*예시:* DAST 도구가 암호화되지 않은 HTTP로 신용카드 정보를 전송하는 서드파티 결제 API 취약점을 발견합니다. 개발자는 모든 API 통신에 HTTPS를 강제합니다.

## 어느 쪽이 더 효과적인가요: SAST vs. DAST?

효과는 다음에 따라 달라집니다:
- SDLC 단계
- 해결하려는 취약점 유형

SAST는 소스 코드 취약점 발견, 비용 절감, 안전한 코딩 촉진을 위한 개발 초기에 가장 효과적입니다. DAST는 프리프로덕션이나 배포된 애플리케이션의 런타임 취약점에 집중해 구성 오류와 안전하지 않은 통합에 대한 인사이트를 제공합니다.

### 하이브리드 접근 방식

하나만 선택하는 대신, **SAST와 DAST를 결합하면 정적 코드와 런타임 취약점을 모두 다루는 다층 보안을 구축**할 수 있습니다. SAST는 개발 초기에 통합되어 컴파일 전 코드 수준 문제를 탐지하고 수정합니다. 테스트 환경에서 실행되면 DAST가 런타임 취약점과 공격 상황에서의 애플리케이션 동작을 식별합니다.

CI/CD 파이프라인에서 둘 다 자동화하면 지속적인 피드백을 제공하고 보안을 유지하면서 개발을 가속화합니다.

애자일이나 DevOps 환경에서는 IAST나 RASP 같은 도구를 추가하면 보안이 더욱 강화됩니다. SAST와 DAST 결과를 상관 분석하면 종합적인 취약점 관리가 가능합니다.

### SAST와 DAST의 대안: RASP

Runtime Application Self-Protection(RASP)은 애플리케이션이 실행되는 서버에 직접 설치하는 고급 보안 솔루션입니다.

RASP는 애플리케이션 런타임 환경에 내장되어 앱 로직과 데이터를 분석합니다:

1. 비정상적인 행동 발생 시 탐지
2. 악의적인 활동 즉시 차단

핵심 차이점: RASP는 외부 도구 없이 위협을 격리하고 해결해 공격을 방지합니다—단순히 경고만 하지 않습니다.

RASP는 SAST와 DAST에 대한 동적 대안입니다. 정적 코드를 분석하는 SAST나 외부 공격을 시뮬레이션하는 DAST와 달리, RASP는 실시간으로 작동해 실행 중 애플리케이션 동작을 모니터링하고 세션 종료나 보안팀 경고를 통해 실시간 위협에 대응합니다.

RASP는 네트워크 방어를 우회하거나 개발 중 놓친 취약점으로부터 애플리케이션을 보호하는 데 특히 효과적입니다.

하지만 RASP에 대한 과신은 조직이 안전한 코딩 관행을 소홀히 하게 만들 수 있습니다. RASP는 런타임 환경에서 작동하므로 애플리케이션 성능에 영향을 줄 수 있습니다. RASP가 근본적인 결함 수정을 대체할 수는 없지만, 수정 기간 동안 지속적인 보호를 제공합니다.

한 RASP 사용자의 말: "RASP 도구 사용 결정은 애플리케이션의 특정 요구사항과 위험 프로필에 대한 철저한 평가를 바탕으로 해야 합니다."

## 미래: SAST, DAST, RASP를 활용한 AppSec 테스팅

애플리케이션 테스팅의 미래는 포괄적인 보안을 위해 SAST, DAST, RASP를 결합하는 것입니다. 보안 과제가 증가하면서 이러한 도구를 CI/CD 파이프라인에 통합하고 프로세스를 자동화하는 것이 필수가 됩니다.

AI와 머신 러닝의 발전으로 오탐을 줄이고 위협 탐지를 개선해 테스팅 효율성이 높아질 것입니다.

### 금융 서비스 산업

금융 부문에서 AI 기반 도구는 취약점 탐지와 완화를 혁신할 것입니다. SAST와 DAST는 개발 및 프리프로덕션 단계에서 코딩 결함과 런타임 문제를 계속 찾아낼 것입니다.

RASP는 AI로 라이브 트랜잭션을 모니터링해 보호를 확장합니다. 예를 들어, RASP는 공격자가 실시간으로 취약점을 악용하는 비정상적인 패턴을 탐지하고, 활동을 차단하면서 보안팀에 알릴 수 있습니다.

### IoT, 엣지, 커넥티드 디바이스

IoT에서는 이러한 도구들이 커넥티드 디바이스를 함께 보호합니다:

- **SAST**는 개발 중 펌웨어에 취약점이 없는지 확인
- **DAST**는 디바이스 간 통신 프로토콜 보안을 테스트
- **RASP**는 배포된 디바이스를 모니터링해 무단 접근이나 새로운 위협으로 인한 데이터 유출을 탐지하고 방지

여러 방법을 사용하면 IoT 생태계의 회복력 유지에 도움이 됩니다. SAST, DAST, RASP를 함께 사용하면 더 강력하고 선제적인 보안 조치로 애플리케이션을 구축하고 배포할 수 있습니다.

## 회복력 있는 보안은 선제적 테스팅에서 시작됩니다

SAST, DAST, RASP를 결합하면 전체 소프트웨어 개발 수명 주기를 아우르는 강력한 보안을 제공합니다.

이러한 방법을 CI/CD 파이프라인에 통합하고 AI 발전을 활용하면 조직이 새로운 위협에 선제적으로 대응할 수 있습니다. 이 방법들이 함께 보안 취약점으로부터 현대 애플리케이션을 보호하는 전체론적 접근 방식을 제공합니다.

## 핵심 요약

- **SAST(정적 분석)**는 소스 코드를 분석해 SDLC 초기에 취약점을 발견하고 비용을 절감합니다
- **DAST(동적 분석)**는 실행 중인 애플리케이션을 테스트해 런타임 취약점과 구성 오류를 식별합니다
- **RASP(런타임 보호)**는 애플리케이션 내부에서 실시간으로 공격을 탐지하고 차단합니다
- 최상의 보안을 위해 SAST, DAST, RASP를 결합한 **다층 방어** 전략을 채택하세요
- CI/CD 파이프라인에 보안 도구를 통합해 **DevSecOps** 실천을 자동화하세요
